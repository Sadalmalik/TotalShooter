# Навигационная система: area-node граф

Решение по задаче "Навигация, восприятие мобов (LOS) и передвижение игрока". Не грузится
автоматически — читать при работе над навигацией/восприятием/движением.

## Концепция

Граф **area-нод** вместо точечных вэйпоинтов (в отличие от классики вроде Amnesia: The Dark
Descent):

- Нода — не точка, а **область** (круг/цилиндр или прямоугольник/параллелепипед с высотой).
  Может покрывать большое пространство — например, длинный коридор — одна прямоугольная
  нода, комната — несколько круглых/квадратных.
- Из любой точки одной ноды можно попасть в любую точку соединённой с ней ноды ("широкая
  дорога"). Узкие переходы между комнатами — через небольшие промежуточные ноды.
- У каждой ноды есть список соседей (граф, не сетка).
- Агенты **не жёстко привязаны** к навигации — двигаются свободно, граф используется как
  подсказка/маршрут, а не как рельсы.
- Ноды — такие же Entity, как и всё остальное в мире: реплицируются/редактируются в sandbox
  тем же механизмом, что и любой другой Entity (см. `.claude/architecture-network.md`,
  "Сетевой статус Entity"), живут в файле мира (`.claude/architecture-core.md`, "World: карта
  как папка").
- **Прыжки/платформы:** соединение между нодами на платформах помечается как "прыжок"; для
  движущихся платформ такое соединение **включается/выключается динамически**, когда ноды
  оказываются достаточно близко друг к другу.

## Почему area-node граф, а не альтернативы

- **Unity NavMesh** — автогенерация и готовый avoidance/steering, но: перестройка дороже для
  частых правок в реалтайме (sandbox-редактирование), не ложится естественно на модель
  "мир = папка с Entity" (навмеш — непрозрачный блоб, не Entity).
- **Униформная чанковая сетка (буквально как в Rampant/Factorio)** — ноль ручной авторской
  работы, тривиальная процедурная генерация, тот же pheromone-алгоритм без адаптации. Но
  проигрывает для закрытых лабиринтов (наш жанр — Alien Shooter/Swarm, не открытая
  местность) — тратит разрешение на сплошные стены, не даёт "широкую дорогу" одним объектом
  для коридора.
- **Точечные вэйпоинты (классика)** — то, от чего сознательно уходим. Главный проигрыш —
  скученность агентов в одной точке при высокой плотности (у нас "сотни-тысячи" врагов);
  area-нода естественно размазывает агентов по пространству вместо схождения в одну XYZ-точку.

**Почему area-node граф удачно ложится на наши задачи (5 причин):**

1. Динамическая перестройка под sandbox-редактирование — ноды это Entity, добавить/убрать/
   перелинковать ноду — дешёвая предсказуемая операция, в отличие от re-bake навмеша.
2. Процедурная генерация уровней из модульных блоков — у каждого блока заранее заданы ноды
   и то, какие из них могут линковаться к соседним блокам. Простая, но мощная система именно
   под жанр закрытых лабиринтов.
3. LOS и звук считаются по графу, не по полной 3D-геометрии (см. раздел ниже).
4. Массовый поиск пути (flow-field/Dijkstra-map) — считается один раз для группы юнитов,
   идущих в одну область, вместо отдельного A* на каждого (см. ниже).
5. Естественная основа для роевого интеллекта (pheromone-подход из Rampant) без отдельной
   chunk-системы — граф нод и есть наша "карта чанков", просто неровная, повторяющая реальную
   проходимую топологию.

## LOS: трёхуровневая эвристика

1. **Отсечение по дистанции** — грубый и дешёвый первый фильтр.
2. **Моб и цель в одной ноде** → видят друг друга (широкая дорога — общая область видимости).
3. **Моб и цель в напрямую связанных (соседних) нодах** → видят друг друга.
4. **Иначе** → явный **2D-плоскостной рейкаст** (narrow-phase, дороже, но выполняется только
   для непростых случаев, отсеянных первыми тремя уровнями). Конкретная реализация рейкаста —
   обсуждается отдельно при переходе к этой задаче реализации (есть готовый опыт/подход).

Граф-связность — не строгий LOS (несоединённые напрямую ноды в большой открытой комнате
технически могут иметь визуальный контакт через третью ноду; связанные соседи могут быть
частично перекрыты динамическими объектами вроде баррикад) — это осознанное приближение
("не RTS, важно ощущение, не точность"), а уровень 4 — уже точная проверка для случаев,
не разрешённых грубой эвристикой.

## Звук: распространение по графу с затуханием

Триггер-звук распространяется по графу нод как заливка с затухающим значением (аналогия —
распространение света в Minecraft). Это **тот же примитив**, что и pheromone-распространение
в Rampant (max среди соседей + decay, см. `.claude/rampant_ai.md`) — один общий механизм
"сигнал с затуханием по графу", переиспользуемый для звука, LOS-broad-phase (опционально) и
роевых сигналов ниже. Не проектировать трижды — один раз, обобщённо.

## Массовый поиск пути (flow-field / Dijkstra-map)

Вместо отдельного A* на каждого юнита — считаем один раз "карту расстояний" от целевой
ноды/области до всех нод, где стоят юниты (аналог flow-field в RTS вроде Supreme Commander).
Любой юнит на любой ноде читает готовый градиент вместо отдельного поиска.

Стыкуется с уже принятым решением по репликации мобов
(`.claude/architecture-network.md`, "Репликация мобов: command-channel и dead reckoning"): AI шлёт
одну команду сразу пачке юнитов, таргет команды — **нода/область**, а не точная мировая
точка. Набросок сетевого пакета:

```
struct MoveCommand
{
    int nodeId;       // целевая область
    short unitCount;
    int unitIds[];    // EntityId юнитов, получающих команду
}
```

## Роевой интеллект: pheromone-сигналы на графе нод

Вместо чанковой сетки Rampant — те же типы сигналов (аналог BASE/PLAYER/DETECTION/RESOURCE/
death-generator) хранятся как метаданные на ноде графа, распространяются тем же примитивом
"сигнал с затуханием по графу" (см. звук выше). Формула подавления сигнала из Rampant
(мультипликативное гашение "защищённых" точек турелями/недавними смертями) переносится на
граф почти без изменений. Прорабатывать детально при дизайне `AIDirector`.

**Важное разделение уровней** (уточнено при дизайне агента): феромоны — это слой **выбора
цели** (goal-selection: "куда вообще идти" — на игрока, на базу, отступать), а flow-field/поиск
пути — слой **исполнения** ("как дойти до уже выбранной цели"). Феромоны решают, какая нода
становится целью поля; поле ведёт юнита к ней. Не смешивать: моб сначала выбирает целевую ноду
по градиенту феромонов (или команде AIDirector), потом катится к ней по flow-field.

## Композиция с уже принятыми решениями

- Граф нод — часть данных World/Entity, реплицируется и редактируется как любой другой Entity.
- Локальный pathfinding на каждом клиенте — по тому же графу (реплицируется вместе с миром),
  агент считает свой путь сам, получая от хоста только редкие команды уровня "иди в
  ноду/область X" — это уже покрыто допуском на неточность между клиентами ("три моба могут
  стоять треугольником на одном клиенте и линией на другом — не важно").

## Реализация: NavNode на Entity (статус)

Расположение: `Assets/TotalShooter/Code/Navigation/NavNode.cs` (рантайм) +
`Assets/TotalShooter/Code/Editor/NavNodeEditor.cs` (инспектор/гизмо/линковка). Черновой
прототип (`NavGridNode`/`NavGrid`/`NavGridAgent`/`TEST`, namespace `Sadalmalik.GridNavigation`)
удалён — переписано на модель "нода = Entity".

**Сделано:**
- **Нода — Entity с компонентом `NavNode`.** Сериализуется/редактируется/(позже)реплицируется
  как любой Entity, живёт в `world.json`.
- **Абстракция формы:** `NavShape` = Circle (radius) / Box (width×length), обе с высотой
  (`Height`). Нода **лежит на полу**: позиция Transform = уровень пола, объём идёт вверх на
  Height (`localY ∈ [0, Height]`; удобно для генерации — класть ноды рейкастом на пол). Позиция и
  поворот влияют на область (масштаб игнорируется — размеры в мировых единицах). `Contains` /
  `ClosestPoint` / `SampleOutline` — общий containment/кламп/контур для обеих форм.
- **Соседи — `NavConnection { int Id; string Type }`** (`List<NavConnection>`): id соседа +
  тип перехода. Тип — **открытая строка** (не enum) ради данных/моддинга; канонические значения
  в `NavConnectionType` (walk/jump/ladder/vent) — подсказки от опечаток. Задел под будущие теги
  переходов и учёт типа в поиске пути (прыжок/лестница/вентиляция). Сериализуются обычным
  `ComponentOverrides` (без спец-хуков в загрузчике). Рантайм-резолв id → (NavNode, тип) через
  реестр мира (`ResolveNeighbours`); в редакторе — обходом сцены. Пока переходы симметричны
  (линковка пишет тип в обе стороны); односторонние — отдельная задача.
- **Редактор:** гизмо объёмной формы (пол/потолок + рёбра), линии соединений, конвексная лента
  прохода между связанными нодами (визуализация "широкой дороги" — выпуклая оболочка контуров).
  Кнопки линковки: по дистанции центров, по соприкосновению форм; ручная правка — список
  Neighbours. Линковка симметрична и при необходимости сама выдаёт стабильные EntityId
  (совместимо со SceneConverter).

**Ещё не сделано (следующие слайсы):**
- **Агент** (`NavAgent` — исполнитель движения врага, парный к `ActionMovement`) + **командир-
  заглушка** (host шлёт "иди в ноду/точку") — ближайший слайс, ради движущихся мобов.
- **Настоящий A\*** — `f = g + h` + приоритетная очередь (прототип был Greedy Best-First без `g`).
- **Движение "куда угодно в пределах ноды"** — стринг-пуллинг поверх списка нод (сейчас есть
  `ClosestPoint`, но агент его ещё не использует), чтобы срезать по диагонали, а не по центрам.
- **Ограничение фазы движения к точной точке** последней нодой (баг прототипа: проход сквозь
  стену/падение с уступа при финальном подходе).
- **Spatial index** под "ближайшая нода" (сейчас резолв/поиск — линейный обход).
- **Region/connected-component ID** — флуд-филл при изменении графа, мгновенный отсев
  недостижимых целей.
- **Wall-aware линковка** — сейчас линковка по дистанции/контакту не проверяет стены между
  нодами (для процедурных блоков связи задаются явно; для ручной расстановки — дыра, следить).

**Оптимизации (позже, не блокеры — сначала рабочая логика):**
- **Тип перехода: строка → int-id + реестр типов.** В горячем цикле поиска пути сравнивать int,
  а не строки. Строка остаётся в данных/JSON (моддинг), резолвится в стабильный int при загрузке
  (тот же приём, что реестр контента; при ≤64 типах — можно бит во флаг-маску под capability-фильтр).
- **Кэш меша гизмо конвекс-лент.** Сейчас `DrawConvexBand` пересобирает выпуклую оболочку каждую
  отрисовку — кэшировать построенный полигон, инвалидировать при изменении графа/формы нод.
- **Веса типов рёбер — конфиг/Lua, не хардкод.** Стоимость перехода по типу (`walk→1, jump→3,
  vent→5…`) хранить как данные (реестр типов), настраиваемые из Lua — ради моддинга. Поиск пути
  берёт вес из таблицы по int-id типа (см. пункт про строку→int выше), не зашивает в код.
- **O(1) доступ Entity по id.** Рантайм уже O(1) (реестр `WorldManager.m_Entities` — словарь,
  `Get(id)`), но: (а) редакторский `NavNodeEditor.FindNode` — полный перебор сцены, завести
  id→NavNode кэш (перестройка по изменению иерархии); (б) в рантайме опц. кэшировать резолвленные
  соседи массивом на ноде (не дёргать словарь каждый кадр в пафайндинге), сбрасывать при правке
  графа. Общий глобальный id→Entity индекс — шаг в сторону ECS.

## Что нужно реализовать/предусмотреть (новые задачи)

- **Абстракция формы ноды** — circle/box (+ высота, цилиндр/параллелепипед), общий
  containment-тест вместо жёстко закодированного `radius`.
- **Region/connected-component ID** — посчитать связные компоненты графа один раз (флуд-филл
  при изменении графа), пометить ноды, мгновенно отклонять поиск пути в недостижимую область.
- **Spatial index под `GetNearestNode`** — рекомендация: простой uniform spatial hash по
  (x, z), а не octree (ноды скорее "поэтажный план", не густо разбросаны в 3D; hash проще в
  реализации и в live-обновлении при runtime-редактировании — не нужен ребаланс дерева).
- **Настоящий A\*** — `f = g + h` с накоплением стоимости пути, приоритетная очередь вместо
  сортировки списка.
- **Движение "куда угодно в пределах ноды"** — стринг-пуллинг/срезание углов поверх
  последовательности нод, вместо хождения к центру каждой.
- **Ограничение `FollowDestination`-фазы** — не выходить за пределы области последней ноды
  пути при движении к точной точке назначения (фикс уже найденного бага).
- **Wall-aware линковка соседей** (или осознанно полагаться на явную авторскую линковку для
  процедурных блоков — решить при реализации).
- **Динамические jump-связи** для движущихся платформ — включение/выключение соединения по
  условию близости нод.
- **LOS: явный 2D-плоскостной рейкаст** (уровень 4 эвристики) — реализация обсуждается
  отдельно при переходе к этой задаче.
- **Обобщённый примитив "сигнал с затуханием по графу"** — переиспользуется для звука,
  LOS-broad-phase (опционально) и pheromone-сигналов роевого ИИ.
- **Массовый поиск пути (flow-field/Dijkstra-map)** — расчёт карты расстояний от целевой
  области до всех нод с юнитами, вместо поюнитового A*.
