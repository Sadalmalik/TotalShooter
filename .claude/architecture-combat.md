## Разрешение одновременных killstreak/убийств: host-validated hits + щедрый assist

Урон и попадания — не самоотчёт клиента. Клиент сообщает хосту "я стрелял по мобу Y в момент
T из позиции P" (для отзывчивости может сразу показать hit-marker локально), хост
отматывает позицию моба на момент T (lag compensation), проверяет правдоподобность
попадания и **сам** считает урон по авторитетным данным оружия/абилки — не доверяя цифре
урона от клиента (иначе это дыра для читов, особенно учитывая PvP-режим).

Если несколько игроков одновременно провели валидный смертельный удар по одному мобу —
раздаём всем "ассист" вместо честного фрага одному. Финализация решения по конкретному
киллу откладывается до истечения окна ожидания = максимальный RTT среди игроков сессии (или
фиксированный запас, ~250-300мс) — не индивидуальный пинг убийцы, чтобы гарантированно
дождаться отчётов всех участников боя.

**Почему:** максимально честно с точки зрения игрока (никого не обделяем в спорной
ситуации), при этом сохраняем авторитетность хоста над цифрами урона/HP.

## Боевая система: hit-registration и lag-compensation в деталях

Детализация механизма из "Разрешение одновременных killstreak/убийств" выше — как именно хост
отматывает позицию цели и проверяет попадание.

**Буфер истории позиций.** На каждую поражаемую цель — кольцевой буфер `{timestamp, position,
rotation}` на хосте (последняя ~1-2 секунды), сэмплируется с частотой **внутреннего тика
симуляции хоста**, не с частотой рассылки клиентам (~1/сек dead reckoning, см.
`architecture-network.md` — это другая, не связанная величина). При проверке попадания по
времени T — линейная интерполяция между двумя соседними сэмплами, реконструирующая позу цели
на момент T.

**Клэмп величины отмотки.** Отмотка ограничена реально измеренным RTT конкретного клиента (+
небольшой запас, с абсолютным потолком) — никогда не доверяем произвольному заявленному
клиентом времени без ограничения (иначе эксплуатируемый вектор чита — "бесконечная
lag-компенсация").

**Отматывается и сам стрелок, не только цель.** Точка вылета выстрела — позиция персонажа
(не камеры — вид от третьего лица сверху, камера и оружие не совпадают, в отличие от FPS) в
момент T, тоже полученная из истории позиций, а не текущая.

**Модель хитбоксов — простой коллайдер (для большинства мобов сфера), не skeletal rewind по
костям.** Это не временная уступка ради скорости реализации, а постоянный выбор, обоснованный
самим ракурсом камеры (сверху и издалека прицельная точность по костям визуально не считывается
игроком так, как в FPS).

**Camera-based видимость может превышать character-based LOS** (камера сверху видит больше,
чем видел бы персонаж с уровня земли за углом) — это ожидаемая, принятая особенность жанра
top-down шутеров, не баг. LOS для геймплейных целей (восприятие мобов и т.п.) считается от
позиции персонажа/моба через граф нод (`.claude/navigation_system.md`), не от камеры.

**Расширение хитбокса на величину dead-reckoning допуска.** Раз клиент и хост в этой
архитектуре в принципе не обязаны соглашаться о точной позиции цели даже без задержки сети
(мы сами приняли допуск на дрифт — см. "Репликация мобов" в `architecture-network.md`) — тест
попадания на хосте использует хитбокс, расширенный на ту же величину допуска, что и dead
reckoning. Это не новая константа — переиспользуем уже принятое число. Рассматривался и более
точный вариант (хост держит отдельную "теневую" проекцию того, что видел именно этот клиент,
для каждого клиент-моб-пары) — отклонён как избыточная инженерия для проекта с приоритетом
"ощущение важнее точности".

**Дробовики — честные отдельные хитсканы на каждую дробинку, с client-side предфильтром.**
Каждая дробинка — отдельный хитскан. Клиент считает попадание каждой дробинки локально по
своей (потенциально смещённой дрифтом) картинке и помечает флагом `hitTest` только те, что
считает попавшими — хост выполняет полную (rewind + intersection) проверку **только** для
помеченных, экономя работу на заведомо мимо пролетевших. **Обязательное условие:** порог
попадания на клиенте при выставлении `hitTest` должен быть **настолько же щедрым**, как
расширенный хитбокс на хосте — иначе клиент может тихо не отправить хосту дробинку, которая
на самом деле должна была засчитаться, что работает против уже принятой философии щедрости
к игроку (ассисты, AoE-margin). Итоговая структура пакета на дробинку:

```
struct
{
    Vector3 origin;
    Vector3 direction;
    bool hitTest;
    int weaponId;   // хост сам считает урон/тип по своим авторитетным данным (через реестр
                     // контента, `architecture-core.md`) — damage/damageType от клиента НЕ
                     // передаются и не используются, чтобы не открывать дыру для читов
}
```

Origin/direction каждого выстрела также переиспользуются как payload чисто косметического
события для рассылки всем клиентам (одинаковый трассер у всех) — одни и те же данные
обслуживают и валидацию, и визуал, дублировать не нужно.

**Проджектайлы — не все становятся Entity.** Простая мгновенная/почти мгновенная пуля —
не Entity, только косметический трассер + мгновенный расчёт на хосте (как хитскан выше).
Снаряды с осмысленным геймплейным состоянием в полёте (граната — таймер, можно поймать;
потенциально ракета — наведение) — Entity со своим `EntityId`. Хост симулирует их полёт
авторитетно и **вперёд в реальном времени**, без отмотки (снаряд не пытается задним числом
доказать попадание в прошлом — он просто летит сейчас). Приятное следствие: раз граната —
Entity, "подобрать недолетевшую гранату" — тот же механизм, что подбор любого лута (см.
"Подбор лута" ниже, эксклюзивно первому) — не долетел, подобрали, не успели кинуть обратно —
взорвалась "в кармане". Не нужно изобретать отдельную механику.

**Взрывы (AoE) — не нужна отмотка вообще.** Точка и момент взрыва уже полностью определены
хостом авторитетно "сейчас" — не заявлены клиентом задним числом. Хост просто делает
пространственный запрос "кто в радиусе R от точки X сейчас" (можно переиспользовать тот же
spatial index — uniform hash по x,z, — что и для `GetNearestNode` в нав-графе) и применяет
урон с затуханием. **Щедрый margin по максимальному пингу игроков** — радиус AoE расширяется
на величину, переведённую из временного запаса (как в разрешении одновременных убийств) в
пространственный через ожидаемую скорость движения игрока — та же логика "лучше нанести урон
тому, кого визуально накрыло на экране, чем оставить его confused невредимым". Точная формула —
на реализацию. Проверка LOS для "не бьёт сквозь стены" — не обязательна для первой версии,
можно добавить позже через тот же граф нод.

**Отложено на потом:** детали формулы урона (фиксированный или рандомизированный, например
диапазон 10-15) — обсуждать при реализации конкретного оружия/абилок.

## Подбор лута: эксклюзивно первому, без щедрости

В отличие от разрешения одновременных убийств (щедрый assist всем), с предметами
поступаем строго: если Item — тот же Entity, при нескольких одновременных заявках на
подбор щедрость привела бы к дублированию предмета, а этого быть не должно. Хост отдаёт
предмет тому, кто подобрал первым; остальные заявки отклоняются. Если игрок ошибся/хочет
поделиться — может просто выбросить предмет обратно.
