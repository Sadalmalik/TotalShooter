# TotalShooter — архитектура: модель данных (Entity, World, реестр контента)

Часть детальных архитектурных решений с обоснованиями ("почему"). Разделено на три файла по
теме — не грузятся автоматически, читать по необходимости для конкретной задачи:
- **`architecture-core.md`** (этот файл) — Entity, World/сохранения, реестр контента,
  скриптинг, компоненты, прогресс игрока, права.
- [`architecture-network.md`](architecture-network.md) — мультиплеер, репликация, сессия, UI.
- [`architecture-combat.md`](architecture-combat.md) — боёвка, hit-registration, лут/килстрики.

Короткая версия без обоснований — `CLAUDE.md` в корне проекта.

## Entity: композиция вместо наследования

Изначальный набросок архитектуры ориентировался на Unreal (`Entity` → `Unit : Entity, IPawn` →
`Item : Entity` и т.д.). От этого отказались: `Entity` — плоский класс без иерархии подтипов.
"Чем является" объект определяется набором прикреплённых к нему компонентов и флагов
(например `CanBeControlled`), а не местом в дереве наследования.

Пример: любой контроллер может управлять любым `Entity` (включая объекты окружения — в
sandbox-режиме это часть веселья), `CanBeControlled` — просто флаг/компонент, а не отдельная
ветка классов.

**Почему:** проще, расширяемее, ближе к тому, как реально нужно (нет искусственных ограничений
кто чем может управлять), и на будущее — этот подход архитектурно совместим с ECS (компонент =
данные, entity = id), что упрощает точечный переход отдельных систем на ECS/Job/Burst позже без
переписывания всей игры.

## Массовые юниты: сначала ООП, точечный ECS/Job/Burst — по факту узких мест

Не берём полноценный DOTS/Entities для геймплейной логики с самого начала. Причины:
- Полный ECS требует переписать логику под data-oriented модель без наследования/виртуальных
  вызовов — это плохо совместимо с runtime-скриптингом (Lua) и sandbox-редактированием
  произвольных объектов (оба сценария завязаны на "обычные" объекты с полями/методами).
- "Сотни-тысячи" мобов не требуют full ECS, чтобы быть производительными — это достижимо
  гибридно: GPU instancing для рендера толпы + точечный `Unity.Jobs`/`Burst` только для
  горячего пути симуляции (движение, LOS/targeting), без перехода на полноценный
  Entities-пакет.

Чтобы будущий перенос конкретной горячей подсистемы на Job/Burst был дешёвым, уже сейчас для
таких подсистем (в первую очередь Movement, Targeting/LOS у мобов) соблюдаем:

1. **Централизованный апдейт вместо `Update()` на каждом объекте.** Горячая логика тикает
   одним менеджером, который раз в кадр проходит по массиву/списку данных всех мобов сразу
   (а не 1000 отдельных `MonoBehaviour.Update()`). Перевод одного такого цикла на
   `IJobParallelFor`/Burst — замена реализации, а не рефакторинг архитектуры.
2. **Данные горячих компонентов — blittable-friendly уже сейчас.** Ссылки на другие
   entity — через id/handle, а не прямые C#-ссылки на объекты (managed-ссылки нельзя
   передать в Job).
3. Всё, что не горячее (Inventory, Ability, Lua-скрипты) — остаётся обычными C#-классами
   навсегда, не оптимизируется.

## Внутриигровой скриптинг: Lua через NLua

Выбран Lua. Библиотека — **NLua** (обёртка над нативной Lua 5.4 C-библиотекой), а не
MoonSharp (чистый C#-интерпретатор). Обычно MoonSharp предпочтителен из-за отсутствия
нативных зависимостей (работает на IL2CPP/мобилках/консолях без боли), но это ограничение
для нас не актуально — таргет платформа Windows, портирование на мобильные/консольные
IL2CPP-платформы в обозримом будущем не планируется. Поэтому берём NLua ради честной
производительности настоящей Lua VM без реальной цены за это в нашем случае.

**Почему Lua вообще:** проверенная десятилетиями культура игрового моддинга именно на
Lua (Garry's Mod, WoW addons, Roblox, и в том числе **Factorio**, чей мод Rampant изучается
в отдельной задаче — приятная синергия: паттерны из Rampant можно будет подсматривать
напрямую на том же языке).

**Архитектура интеграции:**
- Скрипт крутится в `Controller` — то же место, где уже принимает решения AIController
  (см. [[Репликация мобов: command-channel и dead reckoning]] в `architecture-network.md`).
  Пользовательский Lua-скрипт по сути занимает то же архитектурное место, что и обычная
  C#-логика контроллера: решает "что делать" и шлёт команду Entity. Отдельной сетевой модели
  под скрипты не нужно — скрипт исполняется только на хосте (та же авторитетность, что и у
  обычного ИИ), результат реплицируется тем же командным каналом.
- `EntityVariables` — опциональный key-value компонент (blackboard-паттерн) на любом Entity,
  через который скрипты читают/пишут именованные переменные без необходимости заранее знать
  на уровне движка про каждое кастомное поле, которое придумает игрок. На стороне Lua выглядит
  как обращение к полю через индексатор: `someEntity["Keys"] = someEntity["Keys"] + 1`, не как
  вызов геттера/сеттера.
- **По умолчанию `EntityVariables` не сетевой.** Скрипты выполняются только на хосте — значит,
  и переменные для них нужны по-настоящему только там. Если конкретное значение должно быть
  видно клиенту (HP, процент захвата, заряд абилки и т.п.) — оно **явно** "продвигается" в
  репликацию: конкретный именованный ключ конкретного `EntityId` помечается как реплицируемый
  и идёт через тот же кастомный канал, что и остальная репликация Entity (см. [[Сетевой статус
  Entity: EntityId vs NetworkObject]] в `architecture-network.md`). Не синкаем весь
  KV-контейнер целиком — только то, что явно помечено.

**Требование к безопасности (важно, не факультативно):** API, доступный скриптам — **white-list,
не black-list**. Скрипту доступны только явно забинженные игровые функции, никогда — сырые
.NET API (File, Process, Reflection). Это не абстрактная предосторожность: скрипты
физически являются частью шарируемых/скачиваемых миров (см. [[World: карта как папка]] ниже),
то есть игрок будет исполнять у себя чужой недоверенный Lua-код. Дополнительно нужен бюджет
исполнения (лимит инструкций) на скрипт, чтобы `while true do end` не подвесил хост/клиента —
конкретный механизм подбирается при реализации.

## World: карта как папка (аналог сохранений Minecraft / аддонов Gmod)

Конкретизация идеи класса `Map` из `notes.txt` (кастомный формат карт вместо unity-сцен,
чтобы игроки могли сами собирать/редактировать карты). Каждая карта — самодостаточный **мир**
(World), представленный папкой на диске, а не сценой Unity:

```
TotalShooter.exe
Worlds/
  <world-name>/
    world.json      -- актуальное состояние мира (инстансы Entity + оверрайды)
    data.json        -- локальные для этого мира определения контента (prototype-цепочки)
    Scripts/
      *.lua
    (опционально) кастомные ассеты — модели, текстуры и т.п.
```

Игроки, подключаясь к хосту, вместе редактируют мир хоста в реалтайме (через уже принятую
сетевую модель — событийный канал + dead reckoning, ничего дополнительного придумывать не
нужно — см. `architecture-network.md`). Любой игрок может скачать мир к себе локально и
переделывать его самостоятельно (см. "Механизм скачивания/передачи мира" в
`architecture-network.md`).

**Почему:** проверенный паттерн для моддируемых песочниц (Minecraft saves, Gmod
addons/workshop) — папка как самодостаточная единица контента тривиально шарится (скопировал
в `Worlds/` — готово). Напрямую усиливает решение по скриптингу: раз скрипты — файлы внутри
шарируемой папки мира, whitelist-безопасность Lua API — не абстрактная предосторожность, а
конкретное требование (игрок исполняет у себя чужой недоверенный код из скачанного мира).

**Формат файла — JSON** (не бинарный): достаточно для текущего масштаба, читаемо/дифф-friendly,
проще редактировать руками/тулингом на раннем этапе разработки. Пересмотреть, если объём
данных мира когда-нибудь станет реальной проблемой производительности загрузки/сохранения.

**Кастомные ассеты — только через Unity AssetBundles, без рантайм-импорта сырых моделей.**
Не строим отдельный пайплайн рантайм-загрузки произвольных `.fbx`/`.glTF` (например через
glTFast) — осознанно решили не заморачиваться с этим. Новый контент/моды делаются через
AssetBundles (собираются заранее через Editor-пайплайн), адресуются той же строкой-путём,
что и остальные префабы (см. "Реестр контента" ниже). Дороже для конечного игрока-моддера без
Unity (нужно собрать бандл, не просто бросить файл модели в папку), но избавляет от
необходимости писать и поддерживать собственный рантайм-импортёр моделей.

**Мир как прототип для сейвов:** мир-папка одновременно и "шаблон", и "сохранение" — эти
понятия структурно неразличимы, разница только в состоянии. "Начать новую игру из мира X" —
скопировать папку `Worlds/X/` под новым именем и открыть копию; "продолжить" — открыть
существующую папку напрямую. Исходный мир остаётся нетронутым, если явно не открывать его
самого на редактирование. Не требует отдельной архитектуры "шаблон vs сейв" — следует
бесплатно из решения "мир = папка на диске".

**Рантайм-сцена под контент (unity-сцены всё же используем, но иначе).** Мир не грузится как
unity-сцена, но в рантайме удобно иметь одну **пустую сцену-контейнер**: `GameStarter` при старте
создаёт её (`SceneManager.CreateScene` + `SetActiveScene`), и весь инстанцированный контент (мир,
`GameState`, игроки) падает в неё, а бутстрап (менеджеры, NGO-`NetworkManager`, UI) остаётся в
стартовой сцене — чистое разделение. Объекты лежат плоско в корне сцены, **без общего
Transform-родителя** (тысячи детей под одним объектом тормозят Unity). Сцена создаётся локально на
каждом клиенте до сети → гарантированно есть у всех, поэтому **NGO scene management выключаем**
(сцены по сети не синхронизируем — контент едет своим каналом/world-transfer). Unity-сцены при этом
остаются инструментом *авторинга* миров (`SceneConverter` конвертирует сцену → `world.json`) — но
это author-time, не механизм загрузки.

**Версионирование схемы:** `world.json`/`data.json` хранят `"schemaVersion": N`. При
несовпадении с текущей версией движка — явный отказ загрузки с понятной ошибкой, а не тихая
порча/потеря данных (переименованные/переструктурированные поля молча теряются при наивной
загрузке). Полноценные миграции старых версий вперёд (цепочка функций "из N в N+1") —
осознанно отложены: инвестируем в них, когда реально появятся миры, которые жалко терять, а
не заранее. Форма компонентов будет меняться часто в процессе разработки — метка версии здесь
дешёвая страховка от молчаливой поломки, не полноценное решение всех проблем совместимости.

## Иерархия Entity: плоский список + `Parent`, не вложенные `children`

Objects могут быть вложены друг в друга (как Transform-иерархия в Unity — удобно для
модулей уровня при процедурной генерации). Представление в `world.json`/`data.json` —
**плоский список записей, у каждой опциональное поле `Parent` (EntityId родителя)**, не
рекурсивно вложенные `children`.

**Почему:** вся остальная архитектура уже адресует Entity плоско по `EntityId` (репликация,
реестр контента, sandbox-редактирование) — плоская иерархия ничего не ломает и не требует
отдельного кода для обхода дерева. Репарентинг в рантайме (подобрал предмет → стал ребёнком
руки персонажа, построил турель на стене) — частая геймплейная операция, и с плоским
представлением это **та же самая** операция "сменить одно поле у EntityId", что мы уже
спроектировали для sandbox-редактирования (см. "Редактирование в sandbox не-хостом" в
`architecture-network.md`) — не нужен отдельный сетевой примитив "переместить поддерево".
Процедурная генерация не теряет от этого: "модуль" — это набор записей с общим `Parent`/тегом,
копируется тем же способом (набор записей + переприсвоение ID), что и поддерево при вложенном
представлении.

**При загрузке** (`WorldManager`) плоские записи с `Parent` превращаются в настоящую
Unity Transform-иерархию — рекурсивное инстанцирование префабов с вложением через
`Transform.SetParent` в порядке от корней к потомкам. Формат хранения плоский, рантайм-
представление в Unity — обычная иерархия GameObject'ов, с которой смогут нормально работать
и будущие внутриигровые редакторы.

**Обязательное требование реализации:** защита от циклов в цепочке `Parent` (`A.Parent = B`,
`B.Parent = A`) — тот же класс риска и то же решение (отслеживать посещённые `EntityId` при
подъёме по цепочке), что и для циклов в prototype-цепочках реестра контента. Плюс валидация
на "битый" `Parent` (ссылка на несуществующий `EntityId`) — трактовать как отсутствие
родителя, не падать.

## Реестр контента: prototype-цепочки (наследование через префаб + оверрайды)

Единый механизм для двух разных на первый взгляд задач — загрузки состояния мира и описания
кастомных типов контента конкретного мира — это один и тот же приём: **инстанцировать базу
(префаб или другой прототип) и наложить сверху оверрайды полей из JSON**.

- **`world.json`** — инстансы Entity на карте. Каждый инстанс: ссылка на префаб/прототип по
  строковому имени + `Transform` + оверрайды полей конкретных компонентов (сейчас — полный
  дамп текущих значений компонентов при сохранении, не только дельта от дефолтов — проще для
  старта, можно оптимизировать до дельты позже, если понадобится). Служебные поля инстанса:
  `EntityId`, `Parent`, `Name` (логическое имя → в `gameObject.name` при загрузке), `Proto`.
  Детали дампа компонентов (`ComponentOverrides`):
  - Компонент **без полей** сохраняется как `"Comp": {}` — само наличие компонента есть
    состояние (маркеры вроде `SpawnPoint`/`Item` обязаны переживать round-trip).
  - `Proto` для префаб-инстансов проставляет `SceneConverter` автоматически: путь префаба
    относительно `Resources/` без расширения (корни инстансов, только префабы из `Resources`).
  - Встроенный `UnityEngine.Light` дампится точечно (курируемый набор: type/color/intensity/
    range/spotAngle/shadows) и только при наличии — чтобы свет правился из песочницы. Обратно
    читается общим `Apply` (резолв `Light` по имени + запись свойств рефлексией).
- **`data.json`** — локальные для этого мира определения контента, тем же механизмом:
  запись вида `{ "name": "world/ultra-lite-bug", "prototype": "core/lite-bug", ...оверрайды }`
  создаёт новый именованный тип, переиспользуемый (например) в списке `Spawner.Units`.
  Запись с тем же именем, что у существующего прототипа (`"name": "core/lite-bug"`), **заменяет
  его глобально для этого мира** — весь код, спавнящий `"core/lite-bug"` по имени, прозрачно
  получит модифицированную версию. Тот же приём, что используют моды Factorio для
  переопределения ванильных прототипов.
- **Порядок поиска при резолвинге имени** (сейчас и с заделом на будущее): локальные
  определения текущего мира (`data.json`) → моды (когда/если появятся — см. "Заметки на
  будущее") → базовые ассеты, встроенные в игру. Уже готов принять третий слой позже без
  переделки механизма.

**Обязательные свойства реализации:**
- Цепочка прототипов резолвится **рекурсивно на N уровней** (`ultra-lite-bug` →
  `lite-bug` → `core/lite-bug`-префаб), не только на один шаг.
- **Защита от циклов** в цепочке (`A → prototype B → prototype A`) — отслеживать посещённые
  имена при резолвинге, обрывать с ошибкой, а не подвешивать загрузку. Актуально, поскольку
  миры — расшариваемый контент с недоверенными данными (тот же класс риска, что и Lua-скрипты).
- **Единая точка резолвинга имени в объект** (`ResolvePrototype(name)`) — весь код, который
  спавнит контент по имени (Spawner, Ability и т.д.), обязан идти через неё, а не напрямую
  через `Resources.Load`/AssetBundle — иначе локальные оверрайды мира тихо не применятся в
  обход резолвера.

**Известный компромисс:** переопределение существующего имени в `data.json` — мощный приём,
но и footgun: случайное совпадение имени тихо подменит контент везде, где он используется.
Смягчается конвенцией именования по неймспейс-папкам (`core/...` для встроенного контента,
`world/...` для локального для мира) — не структурная защита, а дисциплина. Осознанно
принимаем этот компромисс.

**Адресация префабов (отдельный от реестра контента вопрос):** префабы — визуальные/структурные
Unity-ассеты, адресуются строкой-путём через `Resources.Load`/`AssetBundle`. Неймспейсинг
реализуется структурой папок (`core/...`, `world/...`) — не требует отдельной инфраструктуры,
уже даётся Unity бесплатно. Префаб ≠ определение контента: префаб — это визуал (+ пока что,
на старте — сразу настроенные геймплейные компоненты с дефолтами, см. ниже), определение —
данные о том, как собрать Entity (какие компоненты, какие значения).

**Стартовое решение (осознанно самое простое, не финальное):** префаб сразу несёт настроенные
игромеханические компоненты с дефолтными значениями (не только визуал) — Unity Inspector даёт
это бесплатно, и сейчас контент авторит только Unity-разработчик (не игрок в sandbox). Загрузка
инстанса = `Instantiate(prefab)` + применить оверрайды `world.json`/`data.json` поверх.
**Известное ограничение:** дефолтные значения "запечены" в скомпилированный ассет — игрок без
доступа к Unity Editor не может редактировать "дефолт для всех lite-bug", только оверрайды
конкретных инстансов/локальных прототипов мира. Это осознанный, временный трейд-офф ради
скорости на раннем этапе — при необходимости полноценного sandbox-редактирования "любого"
параметра типа (не только инстанса) эволюционировать к варианту "префаб только визуал + все
игромеханические компоненты и их дефолты — тоже данные в `data.json`", без изменения формата
`world.json`/`data.json` или переделки Spawner — загрузчик уже спроектирован источник-агностично
("собери базу → примени оверрайды").

## Компоненты: Health, Spawner

- **`Health`** — текущее/максимальное здоровье + **типы урона** (по аналогии с Factorio —
  простая модель резистов/множителей по типу урона на цель), достаточно для разной прокачки
  игроков под разные типы врагов. Подсчёт фрагов/assist — не здесь (см. "Разрешение
  одновременных killstreak/убийств" в `architecture-combat.md`), только сама механика
  здоровья/урона.
- **`Spawner`** — компонент на Entity: список тегов (чтобы `AIDirector` мог различать
  спавнеры), список юнитов/префабов на спавн, кулдаун между спавнами, флаг автоспавна, лимит
  спавнов (после которого автоспавн-флаг сбрасывается — точечное управление спавнером
  скриптом/`AIDirector`, например "заход игрока в зону → зональный скрипт командует спавнерам
  заспавнить 20 врагов с интервалом 2 секунды").

## Прогресс игрока: аккаунт (глобально) + профиль в мире (у хоста)

Два разных места хранения:
- **Аккаунт** — файл рядом с игрой на компьютере игрока, глобальные метрики игрока,
  не привязанные к конкретному миру.
- **Профиль в мире** — для каждого игрока в папке конкретного `Worlds/<world-name>/`
  заводится свой файл с достижениями/статистикой именно в этом мире. Хранится у хоста.

Оба файла в той или иной мере доступны скриптам на запись/чтение — например, Lua-логика,
написанная для конкретного мира про океан, может считать сколько игрок проплыл и записать
эту цифру в его профиль в этом мире (а не в глобальный аккаунт).

Точное разделение "что считается глобальным, а что — локальным для мира" пока не
формализовано и будет уточняться по ходу дизайна конкретных фич.

## Права редактирования в sandbox: группы в духе Unix + переключение хостом

Не пытаемся защититься от гриферства/читерства сейчас (проект делается автором и его друзьями,
все — геймдев-программисты/гейм-дизайнеры, играющие друг с другом; защиту решаем только
если/когда проект вырастет за пределы этого круга) — но закладываем модель прав на будущее:
**группы игроков** по аналогии с правами в Unix, что позволяет хранить права на объект в одном
`short` (компактно, не отдельная система per-объект).

Sandbox-режим редактирования включается **хостом**: хост в любой момент может переключить
в режим песочницы себя и опционально других игроков, и обратно. Отдельная защита от
злоупотреблений не нужна — хост это фактически админ сессии, игра не рассчитана на вечно
работающие сервера без присмотра, так что админ всегда на связи.

## Синглплеер — не отдельный случай (подтверждено)

Раз архитектура host-authoritative (см. `architecture-network.md`), синглплеер — это просто
сессия с одним игроком-хостом без подключённых клиентов. Отдельного "оффлайн-режима"
проектировать не нужно.

## Заметки на будущее (не решено, не блокирует текущую работу)

- **Прогрессивная активация областей мира.** Геймплей построен вокруг захвата новых областей
  карты — это естественный механизм ограничить активную симуляцию (незахваченные/нерелевантные
  области могут не думать/не реплицироваться), а не резать дальность симуляции искусственно.
  Стыкуется с решениями по производительности ИИ и по сетевой репликации
  (`architecture-network.md`). Прорабатывать при дизайне AIDirector (после исследования
  Rampant).
- **Мод-система шире, чем скрипты одного мира** (переиспользуемые контент-паки в духе
  Factorio-модов, а не только `Worlds/<name>/Scripts/`) — сознательно не планируется сейчас.
