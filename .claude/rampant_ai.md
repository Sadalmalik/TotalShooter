# Исследование ИИ мода Rampant (RampantFixed 2.02.08) для дизайна AIDirector

Источник: `Research/RampantFixed_2.02.08/` (control.lua, settings.lua, config.lua, data*.lua,
`libs/*.lua`, `prototypes/*`). Версия мода — RampantFixed 2.02.08 (форк оригинального Rampant
для современных Factorio 2.0/Space Age). Документ — выжимка переиспользуемых алгоритмических
идей для `AIDirector` в TotalShooter, не построчный пересказ кода.

Общая метафора мода: карта разбита на chunk'и 32×32, по chunk'ам циркулируют "феромоны"
(числовые метки интереса/угрозы), которые распространяются по принципу **maximum-propagation
with decay** (не честная диффузия/усреднение, а "взять максимум среди соседей и погасить").
Отряды выбирают направление движения, максимизируя взвешенную сумму феромонов текущего/соседних
chunk'ов. Верхнеуровневый "директор" — конечный автомат состояний per-карта (per-планета в
Space Age), с "экономикой очков" (points) как бюджетом на действия и параметром `temperament`
как индикатором обратной связи от игрока.

---

## 1. Алгоритмы ИИ

### 1.1 Феромонная система (`PheromoneUtils.lua`, `ChunkPropertyUtils.lua`, `ScoreChunks.lua`)

Пять типов сигналов, хранящихся per-chunk (индексы вместо строк ради скорости в Lua):

| Феромон | Константа-индекс | Смысл | Источник |
|---|---|---|---|
| `BASE_PHEROMONE` | 1 | "запах вражеской базы" — используется чтобы отряд знал, где дом/куда стягиваться | постройки игрока (обратный эффект — чем больше turret рядом, тем ниже) |
| `PLAYER_PHEROMONE` | 2 | "запах игрока" — куда идти охотиться | физическое присутствие игрока в chunk'е |
| `RESOURCE_PHEROMONE` | 3 | "запах ресурсов" для сеттлеров | количество ресурсных полей на chunk'е |
| `CHUNK_TICK` | 4 | метка "уже обработан в этот тик" (анти-дублирование) | — |
| `BASE_DETECTION_PHEROMONE` | 5 | "насколько сильно игрок обнаружен/раскрыт" — по сути карта тревоги/агро | здания игрока, капается `MAX_BASE_DETECTION_PHEROMONES_IN_CHUNK=10000` |
| death generator (`chunkToDeathGenerator`) | отдельная таблица | "здесь недавно погибли наши" — отпугивающий фактор | смерть юнита ИИ (`deathScent`, `DEATH_PHEROMONE_GENERATOR_AMOUNT=1300`) |
| victory scent (`chunkToVictory`) | отдельная таблица | "здесь недавно выиграли" — снимает death-эффект вокруг точки победы | уничтожение важного здания врага |

**Вход/выход алгоритма распространения** (`processStaticPheromone` — "медленный" проход
BASE/BASE_DETECTION/RESOURCE, `processPheromone` — "быстрый" проход PLAYER):
- Вход: собственное значение chunk'а + максимум среди проходимых соседей (учитывается
  взаимная проходимость по осям через `canMoveChunkDirection`).
- Затухание: `< 500` → линейно `-1`/тик, иначе `× 0.9` (`MOVEMENT_GENERATOR_PERSISTANCE`).
  Для PLAYER_PHEROMONE — передача соседям с коэффициентом `× 0.45` за шаг.
- **Подавление феромона базы турелями/смертями** — ключевой механизм: 
  `turretCount = floor(0.3 * Σturret(8 соседей)) + turret(chunk)`;
  `deathsAsVirtualTurrets = min(floor(0.05 * deathGenerator / 1300), 5)`;
  `multiplier = 0.9 ^ (turretCount + deathsAsVirtualTurrets)`.
  То есть **чем плотнее защищена точка турелями/чем больше там недавно погибло врагов — тем
  меньше "запаха базы" туда идёт**, отряды теряют мотивацию лезть в укреплённые точки.
- Выход: обновлённое числовое значение в chunk'е, читается скорерами (`ScoreChunks.lua`) при
  выборе следующего шага отряда.

**Скоринг направления** (`ScoreChunks.lua`) — не отдельный алгоритм поиска пути, а линейная
комбинация уже посчитанных феромонов конкретного chunk'а, с разными весами под разные "роли":

| Скорер | Формула (упрощённо) | Роль |
|---|---|---|
| `scoreUnitGroupLocation` | `PLAYER − death + BASE + 0.01·DETECTION` | сбор группы к игроку/базе |
| `scoreAttackLocation` | `BASE + 0.01·DETECTION + PLAYER·100` | обычная атака — сильно тянет к игроку |
| `scoreAttackKamikazeLocation` | `DETECTION + PLAYER·100` | камикадзе — бьёт по обнаружению, не по "запаху базы" |
| `scoreResourceLocation` | `−death + RESOURCE − PLAYER·100` | сеттлер-добытчик — избегает игрока |
| `scoreSettlerLocation` | `RESOURCE − death − PLAYER` | новая колония |
| `scoreSiegeLocation` | `0.1·RESOURCE + BASE − 2·death` (0 если DETECTION уже максимум) | осада — не имеет смысла осаждать уже раскрытое |
| `scoreRetreatLocation` | `−DETECTION − death − PLAYER·100 − playerBaseGen·1000` | отступление (в этой версии не используется, см. 1.6) |

Вывод для AIDirector: **не нужен полноценный A*/навмеш для решения "куда толкать врагов"** —
достаточно (а) дискретизировать карту на ячейки, (б) завести 3-4 скалярных "феромона" с простым
правилом max-соседей+decay, (в) считать целевую ячейку как взвешенную сумму этих полей с
разными весами под разные типы поведения. Не Unity NavMesh — это отдельный слой поверх, здесь
речь только про выбор макро-направления/цели.

### 1.2 Chunk-скоринг и обработка карты (`ChunkProcessor.lua`, `MapProcessor.lua`, `ChunkUtils.lua`)

Не единый "скор угрозы", а множество параллельных per-chunk счётчиков (nests/hives/turrets/
traps/utilities игрока и врага), обновляемых через явные `register/unregister` при
постройке/разрушении (`ChunkUtils.accountPlayerEntity`, `registerEnemyBaseStructure`).

**Throttling — центральный архитектурный приём всего мода.** Ничего не обрабатывается по всей
карте за один тик. Паттерны:
- **Round-robin с сохранённым курсором** между вызовами (индекс в массиве или Lua `next()` по
  хэш-таблице) — по N элементов (chunk/отряд/база) за вызов, курсор продолжается со следующего
  вызова с того же места.
- **Фиксированные батчи разного размера под разную "стоимость" операции**: дешёвые сканы
  (проверка одного chunk'а) — 7-20 элементов/тик; дорогие (структурная мутация здания,
  декомпрессия юнитов) — 1-2 элемента/тик.
- **Размазывание по фазам тика**: `pick = tick % 8` — на разные "тики внутри секунды"
  назначены processMap/scanPlayerMap/scanEnemyMap/scanResourceMap/processNests/
  processGrowingBases — не более одной тяжёлой операции на игровой тик.
- **Разные периоды через `on_nth_tick`**: 30 тиков (подземные атаки), 60 (декомпрессия
  посторонних отрядов), 252 (строители), 300 (триггеры demolisher/UI), 3600 (эволюция зданий,
  вероятность подземной атаки), 5000 (обработка demolisher-атак), 30000 (мутации баз,
  vulcanus-эволюция), 36000 (сборка мусора сжатых юнитов).

Вывод для AIDirector: реализовать собственный тикер с "раздачей работы по подсистемам на разные
кадры/тики" вместо одного `Update()`, обрабатывающего всё — то есть то же самое, что и
общее архитектурное решение проекта про централизованные менеджеры вместо Update() на объект,
но применённое ещё и к самому AIDirector внутри себя (не только "не Update на моб", но и
"не пересчитывать весь мир за один тик директора").

### 1.3 Верхнеуровневое планирование — ядро директора (`AIPlanning.lua`, `AIPredicates.lua`)

`aiPlanning.processMapAIs(universe, tick)` — вызывается раз в 60 тиков (раз в секунду),
по 15 карт/планет за вызов. Для каждой активной карты:

**Экономика очков (`points`)** — единственная "валюта" ИИ на все действия:
```
points += min(evo,1)*100 + max(min(retribution,1)*60, activeNests)*2 + min(evo*100, activeRaidNests)*0.1
+ бонус за "непредсказуемость" temperament (от +60 до +180, чем ближе к 0/1 тем больше)
если state == ONSLAUGHT: points *= 2
points *= aiPointsScaler * (прошедшее время / 6 сек) * 0.1
если активно подавление (BiterSupressors уровень ≥ 2): points *= 0.3
points = min(points, maxPoints); overflow cap = maxPoints * 3
```
`maxPoints = max(15500 * evo, 400)`. Очки тратятся на формирование отрядов
(`finalSquadCost≈175·модификатор`), поселенцев (`AI_SETTLER_COST=300`), месть
(`finalVengenceSquadCost≈45·модификатор`). Это прямой аналог "бюджета директора" вроде
Left 4 Dead — накопление ресурса со временем + расход на конкретные "покупки" события.

**`temperament`** (0..1) — метрика обратной связи от действий игрока, считается из отношения
`activeRaidNests / activeNests` (доля "разбуженных" точек среди всех активных вражеских точек):
чем агрессивнее игрок выносит рейдовые гнёзда быстрее, чем ИИ успевает восстанавливать
активность — тем выше `temperament` → тем чаще ИИ выбирает агрессивные состояния
(ONSLAUGHT/RAIDING/SIEGE вместо AGGRESSIVE/MIGRATING). **Это ключевая "адаптивная" часть
challenge-системы** — не таймер, а метрика давления игрока на инфраструктуру ИИ.

**Состояния карты** (`Constants.AI_STATE_*`): `PEACEFUL, AGGRESSIVE, RAIDING, MIGRATING, SIEGE,
ONSLAUGHT, GROWING`. Переход — раз в случайный интервал 10-25 минут (`AI_MIN/MAX_STATE_DURATION`),
через таблицу вероятностей, забинованную по диапазонам `temperament` (низкий → чаще
AGGRESSIVE/MIGRATING, высокий → чаще ONSLAUGHT/RAIDING/SIEGE), дальше — пост-коррекции по
факту ресурсов (например, ONSLAUGHT откатывается в AGGRESSIVE, если `activeNests<70`).
Ночной режим (`aiNocturnalMode`) форсирует переход в SIEGE днём — "враг прячется на свету".

**`aiPredicates.canAttack(map, tick)`** / **`canMigrate(map)`** — простые предикаты-гейты:
атака запрещена в PEACEFUL и в мирном режиме сервера; в AGGRESSIVE — с троттлингом через
`canAttackTick` ("период отдыха" 0.5-3 минуты между волнами); в RAIDING/SIEGE/MIGRATING/
ONSLAUGHT/GROWING — разрешена безусловно. Миграция — только в MIGRATING/SIEGE и с учётом
дня/ночи.

Вывод для AIDirector: паттерн "накопительный бюджет + конечный автомат состояний с
вероятностным переходом, смещаемым метрикой давления игрока" прямо переносим — заменить
`evolution_factor`/`activeNests` на аналоги TotalShooter (число живых баррикад/турелей игроков,
контролируемые зоны, время с начала волны и т.п.).

### 1.4 Формирование атакующих волн (`AIAttackWave.lua`)

- **`formSquads`** — обычная атака: гейты `squadCount < лимит`, `points >= finalSquadCost`,
  случайный ролл `< formSquadThreshold`, у chunk'а должна быть "активность" гнёзд
  (`getNestActiveness>0`). Размер волны — `gaussianRandomRange` вокруг `attackWaveSize`
  (не фиксированное число — рандомизация с отсечкой по границам, чтобы волны не были
  одинаковыми). Есть механизм **дополнительной волны** в состояниях AGGRESSIVE/GROWING:
  `chance = activeNests/15 − squadsGenerated + points/5000`, т.е. чем больше накоплено очков
  и активных точек, тем выше шанс "добавки" сразу вслед за первой волной — имитация "нарастающего
  натиска" без явного отдельного состояния.
- **`formVengenceSquad`** — отряд мести, формируется из накопленной в `rallyUnits`
  "очереди злобы" (`vengenceQueue`), которая наполняется от урона по вражеским точкам с
  кулдауном 10 сек и коэффициентом затухания `chainVengenceCoefficient^vengenceLimiter`
  (защита от спама — чем чаще подряд наносится урон, тем меньше шанс мгновенного ответа).
- **`formSettlers`/`formResourceSettlers`** — экспансия: поиск точки по `RESOURCE_PHEROMONE`,
  ограничение кулдауном на попытку, "избыток очков прожигается" через агрессивное занижение
  `failChance` при `points > 5000/10000`.
- Отряды получают флаги `kamikaze` (не отступает, вероятность растёт с evolution factor:
  `0.10 + evo*0.25`) и `rabid` (3% шанс "бешенства" — вероятно усиленная агрессия).

Вывод: волны — не таймер "каждые N минут отряд из M юнитов", а результат наложения
(бюджет очков) × (вероятностный ролл) × (гейт активности точки) × (рандомизация размера по
Гауссу) — даёт куда менее предсказуемый и не "по расписанию" ощущаемый темп атак, чем
фиксированные волны.

### 1.5 Поведение отрядов — state machine и движение (`SquadAttack.lua`, `MovementUtils.lua`, `UnitGroupUtils.lua`)

Состояния отряда: `GUARDING → RAIDING/SETTLING → (BUILDING при оседании)`, `RETREATING` —
транзитное состояние на один тик диспетчеризации, не постоянный режим.

**`attackMove`** — основной цикл шага атакующего отряда:
1. Если рядом нет ни PLAYER, ни BASE_DETECTION феромона — нечего атаковать, отложить на 10 мин.
2. "Kill streak → settle": если накоплено `kills > 5` и рядом нет игроков — часть отряда
   отпочковывается в сеттлеров новой базы (переиспользование боевого успеха для экспансии).
3. Выбор следующего chunk'а — двухшаговый lookahead (`scoreNeighborsForAttack`): не только
   лучший сосед, но и второй шаг от него — сглаживает путь, движение не идёт "рывками
   chunk-за-chunk".
4. Анти-петля: если один и тот же chunk посещён >4 раз подряд — отряд принудительно
   переключается на `kamikaze` (перестаёт "думать", идёт напролом).
5. Анти-залипание: `checkTick` — если отряд не получал команд 18000 тиков (5 минут) —
   принудительный роспуск.
6. Спид-бафф в "тихих" зонах (низкий DETECTION и PLAYER феромон) — марш ускоряется вне боя.

Вывод: конечный автомат отряда предельно прост (по сути 3 состояния), вся "интеллектуальность"
не в состояниях, а в скоринге направления — состояние лишь выбирает, какой скорер применить.
Anti-stuck на трёх уровнях (таймаут группы, анти-петля с эскалацией в kamikaze, rate-limit
переотправки команд) — важный практический паттерн против "зависших" юнитов у любого
директора с pathfinding через дискретную сетку.

### 1.6 Отступление (`SquadDefense.lua`) — важная оговорка

`retreatUnits` в этой сборке **намеренно отключена** (`error(...)` в начале функции,
единственный вызов закомментирован в `control.lua`). Это не референсный рабочий алгоритм, а
опубликованная, но неисправная/недоделанная заготовка апстрима. Концепт всё равно полезен
как идея (не как код для копирования):
- Кулдаун на chunk (10 сек) + требование "нет вражеских построек на chunk'е" (не отступать
  туда, где ещё есть что защищать).
- Массовая раздача команды одним вызовом движка на весь радиус (`set_multi_command`) вместо
  поштучной раздачи — для Unity-аналога это означает батчинг команд на группу агентов, а не
  вызов на каждого.
- Радиус сбора зависит от важности события: обычный урон — 24, потеря спавнера/ключевого
  здания — 75 (шире зона "паники").
- Поиск уже отступающего отряда поблизости для слияния вместо плодения новых мелких отрядов.

### 1.7 Компрессия юнитов — LOD-паттерн для больших отрядов (`SquadCompression.lua`)

Не AI-алгоритм в смысле принятия решений, а **производительность/perf-хак**, прямо переносимый
на "AIDirector для сотен-тысяч мобов" (то, что явно в TODO проекта). Суть: большой отряд
(>35 юнитов) далеко от игрока схлопывается в 1 представителя на тип юнита + счётчик +
текстовая подпись числа; юниты физически уничтожаются (`entity.destroy()`), информация о
количестве хранится в отдельной таблице. Три уровня по феромонной "температуре" района:

| Состояние | Условие (по PLAYER/DETECTION феромону chunk'а) | Действие |
|---|---|---|
| Полное сжатие | далеко от игрока (`PLAYER<~2.4`) и низкий DETECTION (`<3138`) | 1 юнит на тип + счётчик |
| Мягкое сжатие | средне (`DETECTION<5000`) | распределение по `N` "стекам" вместо 1 на тип (`N` зависит от количества одновременно активных отрядов в мире — глобальный бюджет видимых юнитов) |
| Полная декомпрессия | близко к игроку (`DETECTION>7000` или `PLAYER>~12`) | материализация всех юнитов обратно |

Декомпрессия дробится по кадрам: не более 200 юнитов за один вызов, остаток — в очередь,
досоздание по 50/тик — защита от лаг-спайка при разворачивании тысячного отряда разом.
Материализованные юниты получают кратковременную неуязвимость (1 тик при плановой
декомпрессии, 5 тиков при вынужденной из-за входящего урона) — защита от "смерти в момент
рождения" до того как физика/коллизии успели расставить юнитов по местам.

Прямой вывод для TotalShooter: LOD по числу симулируемых юнитов = агрегированный
"HP-пул + count" для невидимых/далёких отрядов, материализация в реальные Entity только при
приближении, с постепенной (не одномоментной) материализацией и коротким окном неуязвимости
на первый кадр.

### 1.8 Подземные атаки (`UndergroundAttack.lua`)

Механика "нырка": отряд, проигравший бой дальнобойному/артиллерийскому урону (дистанция ≥90
клеток или явно артиллерия), не отступает боем, а "закапывается" — юниты уничтожаются,
взамен создаётся абстрактная запись `undergroundSquad` (без entity в мире), которая по таймеру
(раз в 120 тиков, скорость 5 тайлов/цикл) движется под картой к цели и "выкапывается" рядом с
новой целью (случайная военная постройка / позиция обидчика / общий скоринг chunk'а — выбор
одного из трёх режимов с вероятностями 20/40/40%). По пути идёт сканирование на предмет военных
построек игрока в радиусе 15 — при обнаружении всплытие происходит немедленно рядом с целью,
а не по изначальному плану. Вероятность самого события линейно растёт с evolution factor
(0 при evo≤0.4, максимум settings-значения при evo≥0.8).

Вывод: интересный "антигейт" для боя на истощение — если игрок побеждает исключительно
дальнобойным уроном без риска (артиллерия, снайперки), система создаёт альтернативный вектор
угрозы, который обходит эту тактику (внезапное появление в тылу/у критичной точки).

### 1.9 "Ярость" биттеров (`BitersEnrage.lua`)

Простой локальный бафф: при разрушении вражеского здания игроком, если в радиусе 32 клеток
скопилось >60 живых вражеских юнитов — все получают "стикер ярости" (13 нарастающих уровней,
`speedMultiplier = 1.0 + level*0.10`, максимум ×2.2 на 30 сек), с cooldown на chunk и все
соседние chunk'и (10 сек) от повторного срабатывания. Единственный эффект — скорость движения,
без изменения урона/здоровья. Прогрессия по уровням (не сброс, а накопление при повторных
триггерах, капается на максимуме).

Вывод: дешёвый, но эффектный способ "наказать" игрока, застрявшего в одной точке карты и
выносящего толпы врагов на месте — чем дольше фарм в одной локации, тем быстрее враги.

### 1.10 Подавление ИИ для баланса (`BiterSupressors.lua`)

Билдабельная игроком структура (3 уровня), реагирующая на событие "сектор просканирован"
(радар). Если игрок физически присутствует на планете — подавление принудительно снимается
(механика работает только "за кулисами", не даёт пассивно облегчать активную игру). Эффекты
по уровням: замедление появления demolisher-атак (×1.5 к интервалу, уровень ≥2), заморозка
роста evolution factor (уровень ≥3, откат `set_evolution_factor` назад, если он вырос выше
зафиксированного значения). Активируется на ограниченное время (~1 минута от последнего скана).

Вывод: полезный концепт — "предохранитель" против чрезмерной сложности, управляемый самим
игроком (постройка, а не настройка сложности) и работающий только когда угроза не активна
"на глазах" у игрока (не портит live-бой).

### 1.11 Базы: рост, эволюция, мутация (`BaseUtils.lua`)

- **Тир базы** (0-10) — вычисляется из `evolution_factor` через табличную интерполяцию, но
  с защитой от "вечно низкого evo": принудительный рост тира по времени игры вне зависимости
  от факта эволюции (`min(floor(tick/ticksPerTier), 3)`), чтобы застой в развитии игрока не
  замораживал сложность бесконечно.
- **Alignment/фракция базы** — вероятностный выбор одной или нескольких "фракций" (см. §3) на
  основе `acceptRate` каждой фракции (диапазон тиров, где она допустима, с линейной
  интерполяцией вероятности внутри диапазона). Мутация базы (смена фракции) идёт плавно за
  10-30 "шагов" (интерполяция), а не мгновенно — визуально/геймплейно менее шоковый переход.
- **Рост базы** — новые гнёзда появляются либо форсированно (после урона по базе — временный
  буст), либо естественно при высоком evolution factor (`>0.9`) в чанках с достаточным уровнем
  detection-феромона, но не более 8 гнёзд/ульев или 15 суммарно строений на чанк (защита от
  "стены" построек в одном месте). После 10 неудачных попыток роста подряд — база выключается
  из очереди роста насовсем.
- **Эволюция построек** — здания периодически (раз в 5 часов на весь мир + собственный кулдаун
  на базу) мутируют в более сильный вариант — отдельный от общего evolution factor механизм
  локального усиления конкретных точек.

Вывод: полезная идея "принудительный рост сложности по времени, независимый от прогресса
игрока" — защита от затягивания игры без давления, и "плавная мутация вместо мгновенной смены
типа" — для менее шоковых визуальных апдейтов вражеской фракции на карте.

### 1.12 Паверапы и баланс через evolution factor (`Powerup.lua`)

Не решение директора, а loot-механика, тем не менее показательная как способ **компенсировать
рост сложности прямой наградой игроку**: вражеские спавнеры с шансом (растущим вместе с
evolution factor — от ×1.0 до ×1.6) роняют "камень" с бонусной капсулой (бесконечные патроны +
ваншот биттеров на время). Ваншот-эффект защищён от абьюза на элитных юнитах: цели с
`max_health > 4000` вместо мгновенной смерти получают ×10 урона — компромисс между "весело
косить толпу" и "не убивать боссов одним кликом".

## 2. Взаимосвязь алгоритмов — общий цикл принятия решений

```
on_tick (каждый тик):
├─ removeOneTickImmunity(universe)               — снять неуязвимость с только что декомпрессированных юнитов
├─ processProtectedUnits()
├─ pick = tick % 8  (только одна ветка на тик):
│   ├─ 0: processPendingChunks + processScanChunks + recycleBases + cleanUpMapTables
│   ├─ 1: processPlayers            → обновление PLAYER_PHEROMONE вокруг игроков + "охота" (vengenceQueue)
│   ├─ 2: processMap                → round-robin processPheromone по всей очереди chunk'ов (140/тик)
│   ├─ 3: processStaticMap + disperseVictoryScent  → BASE/BASE_DETECTION/RESOURCE феромоны (20/тик)
│   ├─ 4: scanResourceMap
│   ├─ 5: scanEnemyMap + processSpawners  → ключевая точка ФОРМИРОВАНИЯ ОТРЯДОВ (formSquads/formSettlers/...)
│   ├─ 6: scanPlayerMap + processNests    → processNestActiveness, постановка гнёзд в очередь активности
│   └─ 7: processGrowingBases
├─ if tick % 60 == 0: processMapAIs        → AIPlanning: очки, temperament, смена state автомата
├─ processPendingMutations(universe)        (кроме pick==5)
├─ processActiveNests(universe, tick)       — переоценка активности недавно активных гнёзд (раз в 60 сек/гнездо)
├─ processDecompressQueue(universe)         — досоздание отложенных юнитов (LOD-разворачивание)
└─ processSquads(universe)                  — round-robin по 1 отряду: squadDispatch (state machine) → attackMove/settleMove

on_nth_tick(30):    processUndergroundSquads (движение подземных отрядов)
on_nth_tick(60):    nonRampantCompressedSquads (LOD для чужих юнит-групп)
on_nth_tick(252):   processBuilders / processNonRampantBuilders (достройка осевших сеттлеров)
on_nth_tick(300):   demolisherUtils.processDemolisherTriggers, GUI-обновления
on_nth_tick(3600):  updateUndergroundAttackProbability, evoToBuildingLvl update
on_nth_tick(5000):  demolisherUtils.processDemolishers, suspendClearedMaps
on_nth_tick(30000): setRandomBaseToMutate, vulcanusEvolution
on_nth_tick(36000): checkCompressedUnitsList (сборка мусора)

События (не по таймеру, с собственным cooldown):
- on_entity_died (вражеский юнит/здание убито) → deathScent / victoryScent / rallyUnits / enrageBitersInRange
  / accountPlayerEntity (снятие феромона постройки) / onUnitKilled_DigIn (подземная атака) / points += / checkAndDropPowerup
- on_biter_base_built → createBase / findNearbyBase / registerEnemyBaseStructure
- on_built/mined_entity → accountPlayerEntity (± феромон постройки игрока)
- on_sector_scanned (радар) → processSupression (BiterSupressors)
- on_unit_group_finished_gathering → squadDispatch / processCompression / createUndergroudAttack
```

**Схема причинности верхнего уровня:**

```
Действия игрока (стройка/разрушение/присутствие/урон)
        │
        ▼
Феромонные поля per-chunk (BASE / PLAYER / BASE_DETECTION / RESOURCE / death / victory)
        │  (max-propagation + decay, раз в неск. тиков на chunk через round-robin)
        ▼
Активность гнёзд (processNestActiveness: детекция > порог → +activeNests/+activeRaidNests)
        │
        ▼
AIPlanning: points (бюджет) + temperament (обратная связь от давления игрока)
        │
        ▼
Конечный автомат состояний карты (PEACEFUL/AGGRESSIVE/RAIDING/SIEGE/MIGRATING/ONSLAUGHT/GROWING)
        │
        ▼
AIAttackWave: формирование волн (formSquads/formVengenceSquad/formSettlers), гейты по points+roll+chunk-активности
        │
        ▼
Отряд (squad state machine: GUARDING→RAIDING/SETTLING) + скоринг направления по феромонам (ScoreChunks)
        │
        ▼
Действие в игре (атака/осада/экспансия) → урон по игроку/по базе игрока → замыкает цикл
```

Побочные, относительно независимые контуры, встраивающиеся в общий цикл через свои
события/таймеры, но не блокирующие основной: BitersEnrage (реакция на локальный фарм),
UndergroundAttack (реакция на дальнобойный урон), BiterSupressors/DemolisherUtils
(специфично для Space Age/Vulcanus), Powerup (loot-компенсация сложности),
SquadCompression (сквозной perf-слой, применяется к любому отряду на каждом шаге движения).

## 3. Параметры и типы врагов

### 3.1 Фракции биттеров (`Constants.FACTION_SET`, реально сгенерированные в этой сборке)

Каждая фракция — набор юнитов (biter/spitter/drone) + зданий (spawner×2/turret="worm"/hive),
с сопротивлениями/иммунитетами/атрибутами. `acceptRate = {tierMin, tierMax, rateMin, rateMax}`
определяет диапазон тиров базы, где фракция может появиться, и вероятность.

| Фракция | evo mid | Ключевая механика | Сопротивления/иммунитет | Слабости |
|---|---|---|---|---|
| neutral (базовая) | 0 | ванильные biter/spitter | — | — |
| acid | — | усиленный кислотный урон | +150% acid, +150% poison | — |
| laser | 0.10 | иммунитет к лазеру, overdamage-защита | electric (major), fire/acid (minor), immune laser | physical (minor) |
| electric | 0.10 | атака электричеством/лучом | +150% electric, laser (minor) | — |
| fire | 0.12 | огненная атака (`flame`), поджигает труп | acid (major), physical (minor), immune fire | laser |
| fast | 0.12 | ускоренная атака/движение, long-range immunity 20-25 | explosion/acid (minor) | — |
| physical | 0.12 | высокое HP, "танк" | physical/explosion/acid (major), fire (minor) | laser |
| wasp | 0.20 | дрон-атака (`followsPlayer`), капсульный спавн дронов | — | — |
| spawner (тип) | 0.20 | самовоспроизводящиеся "яйца"/споры (`spawnOnDeath`) | — | — |
| troll | 0.17 | иммунитет к яду, высочайшее HP+регенерация | poison immune, explosion (major), physical (minor) | fire (major) |
| energy-thief | 0.20 | ворует энергию (`drainCrystal`), создаёт crystal-дренаж на электросети | +150% electric, laser (major), poison immune | physical (minor) |
| inferno | 0.20 | экстремальная стойкость к огню/кислоте, `stream`-атака | acid+fire (extreme) | poison (minor), laser (major) |
| poison | — | ядовитое облако при смерти | poison (major), fire/acid (minor) | laser (major), explosion (minor) |
| suicide | 0.35 | взрыв-камикадзе (`bomb`), самоубийство при атаке | fire/acid (major), poison (minor) | explosion (major) |
| nuclear | 0.65 | ядерная атака (`nuclear`), светящийся, самоубийство | fire/acid (major) | explosion (major) |
| juggernaut | 0.75 | "босс"-тип, экстремальная стойкость физ/огонь, poison-облако при смерти | fire/physical (extreme), poison/explosion (major), acid/electric (minor) | laser (minor) |
| arachnids | 0.75 | иммунитет acid+poison, спавн яиц при смерти, "паучья" механика | laser/fire/electric/explosion (minor) | physical (major) |

Все фракции опциональны через `settings.startup["rampantFixed--<type>Enemy"]` (bool),
что уже даёт готовую модель "набора модификаторов сложности игрок может включать/выключать
по вкусу" — прямой аналог для TotalShooter: типы врагов не хардкод, а конфигурируемый набор.

**Масштабирование параметров юнита по тиру** (`prototypes/SwarmUtils.lua` +
`mathUtils.calculateValuesForLevels`): для каждого типа (biter/spitter/drone/worm/spawner/hive)
все числовые атрибуты (health/damage/scale/resistances/spawningTime и т.д.) заданы как массив
из 10 значений (по числу тиров), и `evolutionRequirement = {0, 0.1, 0.2, ..., 0.9}` — то есть
**прямая линейная привязка тира к evolution factor с шагом 0.1**, а внутри тира — линейная
интерполяция (`calculateValuesForLevels`) для дробных состояний. Это уже готовая идея:
не 10 отдельных префабов монстра, а параметрическая кривая характеристик по "уровню
сложности", сэмплируемая в ближайшую дискретную точку либо интерполируемая.

### 3.2 Параметры сложности (`settings.lua`, ключевые)

**Общая сила ИИ:**
- `aiDifficulty` (`Hard`/`Lite`) — грубый переключатель: очки за уничтоженные постройки/спавнеры
  снижаются в разы на `Lite` (см. `RECOVER_NEST_COST`/`RECOVER_WORM_COST` домножение ×0.2/×0.01
  в `control.lua:onDeath`).
- `aiPointsScaler` (0.0-100.0, default 1.0) — прямой множитель накопления `points` (бюджета ИИ).
- `attackWaveMaxSize` (20-600, default 75) и `attackWaveMaxSizeEvoPercent` (5-100%, default 100)
  — максимальный размер волны и evolution-порог, после которого размер уже не растёт.
- `agressiveStart` (bool) — форсированный разгон эволюции по времени в первый час игры
  (`min(evo, min(0.3, 0.1+0.2·tick/216000))`), не давая слишком лёгкое начало на пустой карте.
- `maxNumberOfSquads` (1-30, default 30), `maxNumberOfBuilders` (1-120, default 35) — потолки
  одновременных активных сущностей ИИ (защита производительности хоста/сервера).

**Реактивность на игрока:**
- `attackWaveGenerationUsePlayerProximity` (bool, default true) + `attackPlayerThreshold`
  (default 20) — включает/настраивает чувствительность волн к присутствию игрока рядом.
- `vengenceProbabilityPercent` (default 0.04) и `chainVengenceCoefficient` (default 0.6) —
  базовый шанс "мести" за урон по вражеской точке и коэффициент затухания при повторных ударах
  подряд (защита от спама одной и той же локации).
- `settlementsProbability` (default 0.4) — шанс выбрать миграцию/экспансию вместо атаки.
- `undergroundAttackProbability` (0.01-0.5, default 0.15) — базовая вероятность подземной атаки
  на пике evolution factor.
- `permanentNocturnal` / `allowDaytimePlayerHunting` / `allowDaytimeNonRampantActions` —
  день/ночь как модификатор агрессии (ИИ прячется днём, если включено).

**Per-planet настройки (Space Age)** — отдельная модель сложности на каждую планету/поверхность:
`AI` (выкл/типы поведения, включая demolisher-режим для Vulcanus), `peacePeriod` (часы
гарантированного перемирия от старта), `minEvo` (минимальный evolution factor для активации
ИИ на планете). Показывает паттерн "разные зоны карты — разный профиль сложности", применимый
к разным регионам открытого мира TotalShooter.

**Тонкая балансировка юнитов** (per unit-класс scaler'ы, все default 1.0, диапазон
0.0001-100000): `unitBiterHealthScaler/HealingScaler/SpeedScaler/DamageScaler/RangeScaler` и
аналогичные для Spitter/Drone/Worm/Spawner/Hive — то есть здоровье, урон, скорость и
дальность каждого класса юнитов регулируются независимо, что даёт тонкую настройку "силы"
отдельно от общей эволюции.

**Иммунные механики (относятся к "честности" боя, не к сложности per se)**:
`allowLongRangeImmunity`/`longRangeImmunity_efficiency` (снижение урона от очень дальних атак —
защита от "снайпинга" биттеров без риска) и `allowOneshotProtection`/
`oneshotProtection_efficiency` (кап максимального одиночного урона относительно maxHP) —
готовые идеи защиты AI-юнитов от трюков вроде абьюза дальнобойного оружия без встречного риска.

## 4. Challenge-система — как в целом создаётся ощущение нарастающего вызова

Не единый рычаг "сложность = f(время)", а суперпозиция нескольких независимых контуров
обратной связи, работающих на разных временных масштабах:

1. **Базовый прогресс (evolution factor)** — глобальный, монотонный, управляется убийством
   врагов/загрязнением/временем (ванильный Factorio-механизм, вне мода) — задаёт "потолок"
   характеристик юнитов (health/damage/resistances по тиру) и открывает новые фракции
   (`acceptRate` по диапазону тиров). Долгосрочный, предсказуемый рост.

2. **Локальная реактивность через феромоны** — не про общую сложность, а про "куда именно"
   направлена активность ИИ *прямо сейчас*: чем больше игрок строит/чем ближе стоит — тем выше
   `PLAYER_PHEROMONE`/`BASE_DETECTION_PHEROMONE` в его окрестности, тем чаще туда стягиваются
   волны. Игрок физически не может "спрятаться" бесконечно — активность идёт за его следом,
   но при этом плотная оборона (турели) снижает "запах базы" мультипликативно, давая
   осмысленный контр-плей (турели реально работают, а не только наносят урон, но и снижают
   частоту визитов).

3. **`temperament` — метрика давления игрока на инфраструктуру ИИ**, не на самого игрока.
   Чем эффективнее игрок выносит "активные" (раскрытые/атакующие) вражеские точки быстрее,
   чем ИИ успевает создавать новые — тем сильнее ИИ "разгневан" и переключается в агрессивные
   состояния (ONSLAUGHT/RAIDING/SIEGE). Это ключевой антифраджайл-элемент: **чем лучше игрок
   играет, тем жёстче ответ**, не позволяя закрепить односторонний контроль карты.

4. **Экономика очков (`points`) с капом и "прожиганием избытка"** — сглаживает и одновременно
   усиливает всплески: бюджет копится непрерывно и ограничен потолком (`maxPoints`,
   `overflow cap = maxPoints*3`), но избыток заставляет систему охотнее тратиться (снижение
   `failChance` экспансии при `points>5000/10000`, удвоение начисления в ONSLAUGHT) — то есть
   długое отсутствие давления от игрока не копит "тихую" угрозу бесконечно, а выливается в
   более частые/крупные события.

5. **Точечные "наказания" за конкретные тактики игрока**, а не общий рост цифр:
   - Ярость (BitersEnrage) — наказывает фарм одной точки без движения.
   - Подземные атаки (UndergroundAttack) — наказывает чисто дальнобойный урон без риска
     (артиллерия/снайпинг), пробивая "стену" через альтернативный вектор атаки.
   - Long-range immunity / oneshot protection — прямая защита от абьюза оружия с нулевым
     риском для игрока.
   - Anti-stuck → kamikaze — если ИИ сам застревает (плохой pathfinding), он получает буст
     агрессии вместо того чтобы просто зависнуть — баг pathfinding превращается в фичу
     сложности, а не в очевидный экс-плойт для игрока.

6. **Принудительный рост по времени как страховка от застоя** — тир базы имеет нижнюю границу
   роста по чистому игровому времени (`tierByTime`), независимо от того, копит ли игрок
   evolution factor — устраняет сценарий "нулевого прогресса сложности", если игрок вообще не
   провоцирует эволюцию.

7. **Игрок как соавтор сложности, не только жертва** — параметры сложности разбиты на десятки
   независимых переключателей (типы фракций вкл/выкл, per-planet AI-режим, per-unit-класс
   scaler'ы, safe-buildings список, oneshot/long-range protection), что позволяет очень тонко
   настраивать вызов под конкретную группу игроков без монолитного "уровня сложности 1-10".
   Плюс встроенный "предохранитель" (BiterSupressors), управляемый постройкой в мире, а не
   настройкой меню — то есть баланс "докручивается" через сам геймплей, а не только конфиг.

**Итог для AIDirector TotalShooter**: закладывать не один глобальный слайдер "сложность", а
как минимум три независимых контура — (а) монотонный долгосрочный прогресс (аналог evolution
factor, например от общего времени сессии/суммарного урона по игрокам), (б) локальную
феромон-подобную карту интереса, реагирующую на позицию/постройки/защиту игроков в реальном
времени, (в) метрику давления игрока на инфраструктуру ИИ (аналог temperament) как множитель
агрессии состояний. И отдельно — точечные механики-контрплеи под конкретные "экс-плойт"-паттерны
игры (дальнобойное оружие без риска, фарм одной точки, идеальная турельная стена), а не только
общий рост HP/урона врагов.
