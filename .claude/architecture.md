# TotalShooter — архитектурные решения (детально)

Подробные решения по архитектуре с обоснованиями ("почему"). Короткая версия без обоснований —
в `CLAUDE.md` в корне проекта. Читать этот файл, когда нужен контекст конкретного архитектурного
решения — не грузится автоматически в каждую сессию.

## Entity: композиция вместо наследования

Изначальный набросок архитектуры ориентировался на Unreal (`Entity` → `Unit : Entity, IPawn` →
`Item : Entity` и т.д.). От этого отказались: `Entity` — плоский класс без иерархии подтипов.
"Чем является" объект определяется набором прикреплённых к нему компонентов и флагов
(например `CanBeControlled`), а не местом в дереве наследования.

Пример: любой контроллер может управлять любым `Entity` (включая объекты окружения — в
sandbox-режиме это часть веселья), `CanBeControlled` — просто флаг/компонент, а не отдельная
ветка классов.

**Почему:** проще, расширяемее, ближе к тому, как реально нужно (нет искусственных ограничений
кто чем может управлять), и на будущее — этот подход архитектурно совместим с ECS (компонент =
данные, entity = id), что упрощает точечный переход отдельных систем на ECS/Job/Burst позже без
переписывания всей игры.

## Массовые юниты: сначала ООП, точечный ECS/Job/Burst — по факту узких мест

Не берём полноценный DOTS/Entities для геймплейной логики с самого начала. Причины:
- Полный ECS требует переписать логику под data-oriented модель без наследования/виртуальных
  вызовов — это плохо совместимо с runtime-скриптингом (Lua) и sandbox-редактированием
  произвольных объектов (оба сценария завязаны на "обычные" объекты с полями/методами).
- "Сотни-тысячи" мобов не требуют full ECS, чтобы быть производительными — это достижимо
  гибридно: GPU instancing для рендера толпы + точечный `Unity.Jobs`/`Burst` только для
  горячего пути симуляции (движение, LOS/targeting), без перехода на полноценный
  Entities-пакет.

Чтобы будущий перенос конкретной горячей подсистемы на Job/Burst был дешёвым, уже сейчас для
таких подсистем (в первую очередь Movement, Targeting/LOS у мобов) соблюдаем:

1. **Централизованный апдейт вместо `Update()` на каждом объекте.** Горячая логика тикает
   одним менеджером, который раз в кадр проходит по массиву/списку данных всех мобов сразу
   (а не 1000 отдельных `MonoBehaviour.Update()`). Перевод одного такого цикла на
   `IJobParallelFor`/Burst — замена реализации, а не рефакторинг архитектуры.
2. **Данные горячих компонентов — blittable-friendly уже сейчас.** Ссылки на другие
   entity — через id/handle, а не прямые C#-ссылки на объекты (managed-ссылки нельзя
   передать в Job).
3. Всё, что не горячее (Inventory, Ability, Lua-скрипты) — остаётся обычными C#-классами
   навсегда, не оптимизируется.

## Мультиплеер: Unity Netcode for GameObjects + Unity Relay

Сетевая модель — player-hosted (listen server): один из игроков хост, авторитетно
симулирует игру, остальные подключаются через Unity Relay (NAT traversal, не нужен свой
выделенный сервер и его хостинг). Транспорт/репликация — Unity Netcode for GameObjects (NGO),
официальное и бесплатное решение Unity (платится только фактический трафик Relay).

Кто использует NGO `NetworkObject`, а кто — только наш собственный `EntityId` — см. отдельный
раздел [[Сетевой статус Entity: EntityId vs NetworkObject]] ниже.

**Почему:** ключевая ось масштабирования в этом проекте — не количество *игроков* в сессии
(коопный отряд, обычно немного), а количество *AI-юнитов*. CCU-биллинг сторонних SDK (Photon)
поэтому не является проблемой при таком составе сессий, но NGO+Relay даёт то же самое
бесплатно и official-supported, что проще. Полноценный DOTS Netcode for Entities или Photon
Quantum (deterministic lockstep) дали бы больше "из коробки" под масштаб тысяч юнитов, но
требуют полного перехода геймплейной логики на ECS/детерминированную математику — это
противоречит решению по [[Массовые юниты: сначала ООП]] и требованию к простоте.

**Запасной вариант**, если NGO упрётся в производительность даже при небольшом числе героев/
мета-объектов сессии — **FishNet** (open-source, больше контроля над репликацией). Не начинаем
с него, чтобы не платить сложностью заранее.

## Миграция хоста: осознанно не делаем

При выходе хоста из сессии — **сессия просто завершается**, все отключаются. Никакой
автоматической/бесшовной миграции роли хоста на другого игрока не строим.

**Почему:** хост в нашей архитектуре — единственный источник истины одновременно и по сети
(NGO-сервер), и по геймплейной логике (`GameManager`/`AIController`/`AIDirector`, живые
Lua-скрипты в `Controller`, нереплицированные по умолчанию `EntityVariables`, реестр
`EntityId` в `WorldManager`). У NGO нет встроенной поддержки передачи роли сервера другому
клиенту. Настоящая бесшовная миграция потребовала бы либо реплицировать клиентам заметно
больше состояния, чем нужно для рендера (конфликтует с решением экономить трафик — Lua/
`EntityVariables` намеренно host-only), либо смириться с "мягким рестартом" систем при
передаче роли (Lua-скрипты перезапускаются с нуля, экономика AIDirector обнуляется).
Даже Minecraft (наш референс для формата мира) не решает эту проблему в ванильном
LAN-хостинге — просто требует быстрого рестарта сервера.

**Восстановление на практике** — не автоматическое, а ручное: кто-то из игроков поднимает
новый хостинг из последнего автосейва мира (уже решённая механика), при необходимости получив
файл мира от других вручную (уже решено — [[Механизм скачивания/передачи мира]], миры
передаются вручную). Никакой новой инженерии не требуется.

**Известное ограничение:** автосейв пишется в папку мира на диске хоста, а не рассылается
остальным игрокам автоматически — если хост стал недоступен без предупреждения, у других
игроков может не быть самой свежей копии мира (только та, что скачивали раньше). Приемлемо
для игры с друзьями; пересмотреть, если это когда-нибудь станет реальной проблемой.

## Сетевой статус Entity: EntityId (наш) vs NetworkObject (NGO)

Единое правило для всех `Entity`, без исключений:

- **У каждого `Entity` есть собственный `EntityId`** (наш ID, не NGO-шный `NetworkObjectId`).
  Это база, есть всегда. По нему адресуются команды/события в кастомном канале репликации
  (см. [[Репликация мобов: command-channel и dead reckoning]]) и Lua-скрипты обращаются к
  конкретному Entity.
- **`NetworkObject` — опциональная надстройка NGO**, вешается только на то, что реально
  нуждается в индивидуальном авторитетном владении через NGO: **герои игроков** и
  **мета-объекты сессии** (`GameState`, `PlayerState`, `PlayerController`).
- **Всё остальное в игровом мире — просто `Entity` с `EntityId`, без `NetworkObject`**: мобы,
  турели/баррикады/ловушки/мины, лут, снаряды. Реплицируется единым кастомным каналом
  (событие спавна + редкие апдейты состояния), независимо от того, "штучный" это объект или
  один из тысячи.

**Почему так, а не "важные штучные объекты — NetworkObject, толпа — нет":** турели/баррикады
не двигаются — им не нужен даже dead reckoning, только событие спавна + редкие апдейты
HP/состояния. Они попадают в ту же машинерию, что и мобы, практически бесплатно, раз мы её
всё равно строим для мобов. Не плодим два параллельных механизма репликации ("важное штучное"
и "толпа") — один кастомный канал для всего в игровом мире, кроме героев, и NGO только для
героев + мета-объектов сессии.

## Репликация мобов: command-channel + dead reckoning вместо постоянного снапшота позиции

Движение/pathfinding моба — **локальная логика**, исполняется независимо на каждом клиенте
(и на хосте). AI на хосте не решает "куда шагнуть в этом кадре" — он раз в 200-500мс (см.
[[Массовые юниты: сначала ООП]]) принимает решение уровня "иди в точку X" / "атакуй Y" /
"используй абилку Z" и рассылает это как **команду** через reliable event-канал (тот же,
что для гранат/хила/смерти). Моб на клиенте сам исполняет команду (свой локальный
pathfinding/steering), а не ждёт постоянных апдейтов позиции.

Поверх этого — редкая (~1 раз/сек) авторитетная позиция с хоста для коррекции дрифта
(dead reckoning / client-side extrapolation + periodic reconciliation, стандартный паттерн
для реплицируемых NPC/техники в шутерах). Коррекция — только если расхождение вышло за
допустимый радиус погрешности (не дёргать позицию за каждую мелочь). Точный алгоритм
коррекции — при реализации.

**Почему:** это не RTS, точная синхронизация мобов между клиентами не нужна (три моба могут
стоять треугольником на одном клиенте и линией на другом — не важно). Важно ощущение
"игра работает правильно", а не побитовое совпадение. Такой подход резко снижает сетевой
трафик по сравнению с постоянной трансляцией позиции/направления каждого моба, и совпадает
с уже выделенной "горячей" подсистемой движения (тот же `MovementManager` — кандидат на
Job/Burst позже).

Нюанс на будущее (не сейчас, территория задачи навигации): если карта редактируется в
реалтайме (sandbox), локальный pathfinding клиента должен видеть те же изменения навигации,
что и хост.

## Разрешение одновременных killstreak/убийств: host-validated hits + щедрый assist

Урон и попадания — не самоотчёт клиента. Клиент сообщает хосту "я стрелял по мобу Y в момент
T из позиции P" (для отзывчивости может сразу показать hit-marker локально), хост
отматывает позицию моба на момент T (lag compensation), проверяет правдоподобность
попадания и **сам** считает урон по авторитетным данным оружия/абилки — не доверяя цифре
урона от клиента (иначе это дыра для читов, особенно учитывая PvP-режим).

Если несколько игроков одновременно провели валидный смертельный удар по одному мобу —
раздаём всем "ассист" вместо честного фрага одному. Финализация решения по конкретному
киллу откладывается до истечения окна ожидания = максимальный RTT среди игроков сессии (или
фиксированный запас, ~250-300мс) — не индивидуальный пинг убийцы, чтобы гарантированно
дождаться отчётов всех участников боя.

**Почему:** максимально честно с точки зрения игрока (никого не обделяем в спорной
ситуации), при этом сохраняем авторитетность хоста над цифрами урона/HP.

## Боевая система: hit-registration и lag-compensation в деталях

Детализация механизма из [[Разрешение одновременных killstreak/убийств]] — как именно хост
отматывает позицию цели и проверяет попадание.

**Буфер истории позиций.** На каждую поражаемую цель — кольцевой буфер `{timestamp, position,
rotation}` на хосте (последняя ~1-2 секунды), сэмплируется с частотой **внутреннего тика
симуляции хоста**, не с частотой рассылки клиентам (~1/сек dead reckoning — это другая,
не связанная величина). При проверке попадания по времени T — линейная интерполяция между
двумя соседними сэмплами, реконструирующая позу цели на момент T.

**Клэмп величины отмотки.** Отмотка ограничена реально измеренным RTT конкретного клиента (+
небольшой запас, с абсолютным потолком) — никогда не доверяем произвольному заявленному
клиентом времени без ограничения (иначе эксплуатируемый вектор чита — "бесконечная
lag-компенсация").

**Отматывается и сам стрелок, не только цель.** Точка вылета выстрела — позиция персонажа
(не камеры — вид от третьего лица сверху, камера и оружие не совпадают, в отличие от FPS) в
момент T, тоже полученная из истории позиций, а не текущая.

**Модель хитбоксов — простой коллайдер (для большинства мобов сфера), не skeletal rewind по
костям.** Это не временная уступка ради скорости реализации, а постоянный выбор, обоснованный
самим ракурсом камеры (сверху и издалека прицельная точность по костям визуально не считывается
игроком так, как в FPS).

**Camera-based видимость может превышать character-based LOS** (камера сверху видит больше,
чем видел бы персонаж с уровня земли за углом) — это ожидаемая, принятая особенность жанра
top-down шутеров, не баг. LOS для геймплейных целей (восприятие мобов и т.п.) считается от
позиции персонажа/моба через граф нод ([[Nav-граф]], `.claude/navigation_system.md`), не от
камеры.

**Расширение хитбокса на величину dead-reckoning допуска.** Раз клиент и хост в этой
архитектуре в принципе не обязаны соглашаться о точной позиции цели даже без задержки сети
(мы сами приняли допуск на дрифт в [[Репликация мобов]]) — тест попадания на хосте использует
хитбокс, расширенный на ту же величину допуска, что и dead reckoning. Это не новая
константа — переиспользуем уже принятое число. Рассматривался и более точный вариант (хост
держит отдельную "теневую" проекцию того, что видел именно этот клиент, для каждого
клиент-моб-пары) — отклонён как избыточная инженерия для проекта с приоритетом "ощущение
важнее точности".

**Дробовики — честные отдельные хитсканы на каждую дробинку, с client-side предфильтром.**
Каждая дробинка — отдельный хитскан. Клиент считает попадание каждой дробинки локально по
своей (потенциально смещённой дрифтом) картинке и помечает флагом `hitTest` только те, что
считает попавшими — хост выполняет полную (rewind + intersection) проверку **только** для
помеченных, экономя работу на заведомо мимо пролетевших. **Обязательное условие:** порог
попадания на клиенте при выставлении `hitTest` должен быть **настолько же щедрым**, как
расширенный хитбокс на хосте — иначе клиент может тихо не отправить хосту дробинку, которая
на самом деле должна была засчитаться, что работает против уже принятой философии щедрости
к игроку (ассисты, AoE-margin). Итоговая структура пакета на дробинку:

```
struct
{
    Vector3 origin;
    Vector3 direction;
    bool hitTest;
    int weaponId;   // хост сам считает урон/тип по своим авторитетным данным (через реестр
                     // контента, [[Реестр контента]]) — damage/damageType от клиента НЕ
                     // передаются и не используются, чтобы не открывать дыру для читов
}
```

Origin/direction каждого выстрела также переиспользуются как payload чисто косметического
события для рассылки всем клиентам (одинаковый трассер у всех) — одни и те же данные
обслуживают и валидацию, и визуал, дублировать не нужно.

**Проджектайлы — не все становятся Entity.** Простая мгновенная/почти мгновенная пуля —
не Entity, только косметический трассер + мгновенный расчёт на хосте (как хитскан выше).
Снаряды с осмысленным геймплейным состоянием в полёте (граната — таймер, можно поймать;
потенциально ракета — наведение) — Entity со своим `EntityId`. Хост симулирует их полёт
авторитетно и **вперёд в реальном времени**, без отмотки (снаряд не пытается задним числом
доказать попадание в прошлом — он просто летит сейчас). Приятное следствие: раз граната —
Entity, "подобрать недолетевшую гранату" — тот же механизм, что подбор любого лута
([[Подбор лута]], эксклюзивно первому) — не долетел, подобрали, не успели кинуть обратно —
взорвалась "в кармане". Не нужно изобретать отдельную механику.

**Взрывы (AoE) — не нужна отмотка вообще.** Точка и момент взрыва уже полностью определены
хостом авторитетно "сейчас" — не заявлены клиентом задним числом. Хост просто делает
пространственный запрос "кто в радиусе R от точки X сейчас" (можно переиспользовать тот же
spatial index — uniform hash по x,z, — что и для `GetNearestNode` в нав-графе) и применяет
урон с затуханием. **Щедрый margin по максимальному пингу игроков** — радиус AoE расширяется
на величину, переведённую из временного запаса (как в разрешении одновременных убийств) в
пространственный через ожидаемую скорость движения игрока — та же логика "лучше нанести урон
тому, кого визуально накрыло на экране, чем оставить его confused невредимым". Точная формула —
на реализацию. Проверка LOS для "не бьёт сквозь стены" — не обязательна для первой версии,
можно добавить позже через тот же граф нод.

**Отложено на потом:** детали формулы урона (фиксированный или рандомизированный, например
диапазон 10-15) — обсуждать при реализации конкретного оружия/абилок.

## Подбор лута: эксклюзивно первому, без щедрости

В отличие от разрешения одновременных убийств (щедрый assist всем), с предметами
поступаем строго: если Item — тот же Entity, при нескольких одновременных заявках на
подбор щедрость привела бы к дублированию предмета, а этого быть не должно. Хост отдаёт
предмет тому, кто подобрал первым; остальные заявки отклоняются. Если игрок ошибся/хочет
поделиться — может просто выбросить предмет обратно.

## Внутриигровой скриптинг: Lua через NLua

Выбран Lua. Библиотека — **NLua** (обёртка над нативной Lua 5.4 C-библиотекой), а не
MoonSharp (чистый C#-интерпретатор). Обычно MoonSharp предпочтителен из-за отсутствия
нативных зависимостей (работает на IL2CPP/мобилках/консолях без боли), но это ограничение
для нас не актуально — таргет платформа Windows, портирование на мобильные/консольные
IL2CPP-платформы в обозримом будущем не планируется. Поэтому берём NLua ради честной
производительности настоящей Lua VM без реальной цены за это в нашем случае.

**Почему Lua вообще:** проверенная десятилетиями культура игрового моддинга именно на
Lua (Garry's Mod, WoW addons, Roblox, и в том числе **Factorio**, чей мод Rampant изучается
в отдельной задаче — приятная синергия: паттерны из Rampant можно будет подсматривать
напрямую на том же языке).

**Архитектура интеграции:**
- Скрипт крутится в `Controller` — то же место, где уже принимает решения AIController
  (см. [[Репликация мобов: command-channel и dead reckoning]]). Пользовательский Lua-скрипт
  по сути занимает то же архитектурное место, что и обычная C#-логика контроллера: решает
  "что делать" и шлёт команду Entity. Отдельной сетевой модели под скрипты не нужно — скрипт
  исполняется только на хосте (та же авторитетность, что и у обычного ИИ), результат
  реплицируется тем же командным каналом.
- `EntityVariables` — опциональный key-value компонент (blackboard-паттерн) на любом Entity,
  через который скрипты читают/пишут именованные переменные без необходимости заранее знать
  на уровне движка про каждое кастомное поле, которое придумает игрок. На стороне Lua выглядит
  как обращение к полю через индексатор: `someEntity["Keys"] = someEntity["Keys"] + 1`, не как
  вызов геттера/сеттера.
- **По умолчанию `EntityVariables` не сетевой.** Скрипты выполняются только на хосте — значит,
  и переменные для них нужны по-настоящему только там. Если конкретное значение должно быть
  видно клиенту (HP, процент захвата, заряд абилки и т.п.) — оно **явно** "продвигается" в
  репликацию: конкретный именованный ключ конкретного `EntityId` помечается как реплицируемый
  и идёт через тот же кастомный канал, что и остальная репликация Entity (см. [[Сетевой статус
  Entity: EntityId vs NetworkObject]]). Не синкаем весь KV-контейнер целиком — только то, что
  явно помечено.

**Требование к безопасности (важно, не факультативно):** API, доступный скриптам — **white-list,
не black-list**. Скрипту доступны только явно забинженные игровые функции, никогда — сырые
.NET API (File, Process, Reflection). Это не абстрактная предосторожность: скрипты
физически являются частью шарируемых/скачиваемых миров (см. [[World: карта как папка]]),
то есть игрок будет исполнять у себя чужой недоверенный Lua-код. Дополнительно нужен бюджет
исполнения (лимит инструкций) на скрипт, чтобы `while true do end` не подвесил хост/клиента —
конкретный механизм подбирается при реализации.

## World: карта как папка (аналог сохранений Minecraft / аддонов Gmod)

Конкретизация идеи класса `Map` из `notes.txt` (кастомный формат карт вместо unity-сцен,
чтобы игроки могли сами собирать/редактировать карты). Каждая карта — самодостаточный **мир**
(World), представленный папкой на диске, а не сценой Unity:

```
TotalShooter.exe
Worlds/
  <world-name>/
    world.json      -- актуальное состояние мира (инстансы Entity + оверрайды)
    data.json        -- локальные для этого мира определения контента (prototype-цепочки)
    Scripts/
      *.lua
    (опционально) кастомные ассеты — модели, текстуры и т.п.
```

Игроки, подключаясь к хосту, вместе редактируют мир хоста в реалтайме (через уже принятую
сетевую модель — событийный канал + dead reckoning, ничего дополнительного придумывать не
нужно). Любой игрок может скачать мир к себе локально и переделывать его самостоятельно.

**Почему:** проверенный паттерн для моддируемых песочниц (Minecraft saves, Gmod
addons/workshop) — папка как самодостаточная единица контента тривиально шарится (скопировал
в `Worlds/` — готово). Напрямую усиливает решение по скриптингу: раз скрипты — файлы внутри
шарируемой папки мира, whitelist-безопасность Lua API — не абстрактная предосторожность, а
конкретное требование (игрок исполняет у себя чужой недоверенный код из скачанного мира).

**Формат файла — JSON** (не бинарный): достаточно для текущего масштаба, читаемо/дифф-friendly,
проще редактировать руками/тулингом на раннем этапе разработки. Пересмотреть, если объём
данных мира когда-нибудь станет реальной проблемой производительности загрузки/сохранения.

**Мир как прототип для сейвов:** мир-папка одновременно и "шаблон", и "сохранение" — эти
понятия структурно неразличимы, разница только в состоянии. "Начать новую игру из мира X" —
скопировать папку `Worlds/X/` под новым именем и открыть копию; "продолжить" — открыть
существующую папку напрямую. Исходный мир остаётся нетронутым, если явно не открывать его
самого на редактирование. Не требует отдельной архитектуры "шаблон vs сейв" — следует
бесплатно из решения "мир = папка на диске".

**Версионирование схемы:** `world.json`/`data.json` хранят `"schemaVersion": N`. При
несовпадении с текущей версией движка — явный отказ загрузки с понятной ошибкой, а не тихая
порча/потеря данных (переименованные/переструктурированные поля молча теряются при наивной
загрузке). Полноценные миграции старых версий вперёд (цепочка функций "из N в N+1") —
осознанно отложены: инвестируем в них, когда реально появятся миры, которые жалко терять, а
не заранее. Форма компонентов будет меняться часто в процессе разработки — метка версии здесь
дешёвая страховка от молчаливой поломки, не полноценное решение всех проблем совместимости.

## Иерархия Entity: плоский список + `Parent`, не вложенные `children`

Objects могут быть вложены друг в друга (как Transform-иерархия в Unity — удобно для
модулей уровня при процедурной генерации). Представление в `world.json`/`data.json` —
**плоский список записей, у каждой опциональное поле `Parent` (EntityId родителя)**, не
рекурсивно вложенные `children`.

**Почему:** вся остальная архитектура уже адресует Entity плоско по `EntityId` (репликация,
реестр контента, sandbox-редактирование) — плоская иерархия ничего не ломает и не требует
отдельного кода для обхода дерева. Репарентинг в рантайме (подобрал предмет → стал ребёнком
руки персонажа, построил турель на стене) — частая геймплейная операция, и с плоским
представлением это **та же самая** операция "сменить одно поле у EntityId", что мы уже
спроектировали для sandbox-редактирования ([[Sandbox-редактирование не-хостом]]) — не нужен
отдельный сетевой примитив "переместить поддерево". Процедурная генерация не теряет от этого:
"модуль" — это набор записей с общим `Parent`/тегом, копируется тем же способом (набор записей
+ переприсвоение ID), что и поддерево при вложенном представлении.

**При загрузке** (`WorldManager`) плоские записи с `Parent` превращаются в настоящую
Unity Transform-иерархию — рекурсивное инстанцирование префабов с вложением через
`Transform.SetParent` в порядке от корней к потомкам. Формат хранения плоский, рантайм-
представление в Unity — обычная иерархия GameObject'ов, с которой смогут нормально работать
и будущие внутриигровые редакторы.

**Обязательное требование реализации:** защита от циклов в цепочке `Parent` (`A.Parent = B`,
`B.Parent = A`) — тот же класс риска и то же решение (отслеживать посещённые `EntityId` при
подъёме по цепочке), что и для циклов в prototype-цепочках реестра контента. Плюс валидация
на "битый" `Parent` (ссылка на несуществующий `EntityId`) — трактовать как отсутствие
родителя, не падать.

## Реестр контента: prototype-цепочки (наследование через префаб + оверрайды)

Единый механизм для двух разных на первый взгляд задач — загрузки состояния мира и описания
кастомных типов контента конкретного мира — это один и тот же приём: **инстанцировать базу
(префаб или другой прототип) и наложить сверху оверрайды полей из JSON**.

- **`world.json`** — инстансы Entity на карте. Каждый инстанс: ссылка на префаб/прототип по
  строковому имени + `Transform` + оверрайды полей конкретных компонентов (сейчас — полный
  дамп текущих значений компонентов при сохранении, не только дельта от дефолтов — проще для
  старта, можно оптимизировать до дельты позже, если понадобится).
- **`data.json`** — локальные для этого мира определения контента, тем же механизмом:
  запись вида `{ "name": "world/ultra-lite-bug", "prototype": "core/lite-bug", ...оверрайды }`
  создаёт новый именованный тип, переиспользуемый (например) в списке `Spawner.Units`.
  Запись с тем же именем, что у существующего прототипа (`"name": "core/lite-bug"`), **заменяет
  его глобально для этого мира** — весь код, спавнящий `"core/lite-bug"` по имени, прозрачно
  получит модифицированную версию. Тот же приём, что используют моды Factorio для
  переопределения ванильных прототипов.
- **Порядок поиска при резолвинге имени** (сейчас и с заделом на будущее): локальные
  определения текущего мира (`data.json`) → моды (когда/если появятся — см. "Заметки на
  будущее") → базовые ассеты, встроенные в игру. Уже готов принять третий слой позже без
  переделки механизма.

**Обязательные свойства реализации:**
- Цепочка прототипов резолвится **рекурсивно на N уровней** (`ultra-lite-bug` →
  `lite-bug` → `core/lite-bug`-префаб), не только на один шаг.
- **Защита от циклов** в цепочке (`A → prototype B → prototype A`) — отслеживать посещённые
  имена при резолвинге, обрывать с ошибкой, а не подвешивать загрузку. Актуально, поскольку
  миры — расшариваемый контент с недоверенными данными (тот же класс риска, что и Lua-скрипты).
- **Единая точка резолвинга имени в объект** (`ResolvePrototype(name)`) — весь код, который
  спавнит контент по имени (Spawner, Ability и т.д.), обязан идти через неё, а не напрямую
  через `Resources.Load`/AssetBundle — иначе локальные оверрайды мира тихо не применятся в
  обход резолвера.

**Известный компромисс:** переопределение существующего имени в `data.json` — мощный приём,
но и footgun: случайное совпадение имени тихо подменит контент везде, где он используется.
Смягчается конвенцией именования по неймспейс-папкам (`core/...` для встроенного контента,
`world/...` для локального для мира) — не структурная защита, а дисциплина. Осознанно
принимаем этот компромисс.

**Адресация префабов (отдельный от реестра контента вопрос):** префабы — визуальные/структурные
Unity-ассеты, адресуются строкой-путём через `Resources.Load`/`AssetBundle`. Неймспейсинг
реализуется структурой папок (`core/...`, `world/...`) — не требует отдельной инфраструктуры,
уже даётся Unity бесплатно. Префаб ≠ определение контента: префаб — это визуал (+ пока что,
на старте — сразу настроенные геймплейные компоненты с дефолтами, см. ниже), определение —
данные о том, как собрать Entity (какие компоненты, какие значения).

**Стартовое решение (осознанно самое простое, не финальное):** префаб сразу несёт настроенные
игромеханические компоненты с дефолтными значениями (не только визуал) — Unity Inspector даёт
это бесплатно, и сейчас контент авторит только Unity-разработчик (не игрок в sandbox). Загрузка
инстанса = `Instantiate(prefab)` + применить оверрайды `world.json`/`data.json` поверх.
**Известное ограничение:** дефолтные значения "запечены" в скомпилированный ассет — игрок без
доступа к Unity Editor не может редактировать "дефолт для всех lite-bug", только оверрайды
конкретных инстансов/локальных прототипов мира. Это осознанный, временный трейд-офф ради
скорости на раннем этапе — при необходимости полноценного sandbox-редактирования "любого"
параметра типа (не только инстанса) эволюционировать к варианту "префаб только визуал + все
игромеханические компоненты и их дефолты — тоже данные в `data.json`", без изменения формата
`world.json`/`data.json` или переделки Spawner — загрузчик уже спроектирован источник-агностично
("собери базу → примени оверрайды").

## Прогресс игрока: аккаунт (глобально) + профиль в мире (у хоста)

Два разных места хранения:
- **Аккаунт** — файл рядом с игрой на компьютере игрока, глобальные метрики игрока,
  не привязанные к конкретному миру.
- **Профиль в мире** — для каждого игрока в папке конкретного `Worlds/<world-name>/`
  заводится свой файл с достижениями/статистикой именно в этом мире. Хранится у хоста.

Оба файла в той или иной мере доступны скриптам на запись/чтение — например, Lua-логика,
написанная для конкретного мира про океан, может считать сколько игрок проплыл и записать
эту цифру в его профиль в этом мире (а не в глобальный аккаунт).

Точное разделение "что считается глобальным, а что — локальным для мира" пока не
формализовано и будет уточняться по ходу дизайна конкретных фич.

## Права редактирования в sandbox: группы в духе Unix + переключение хостом

Не пытаемся защититься от гриферства/читерства сейчас (проект делается автором и его друзьями,
все — геймдев-программисты/гейм-дизайнеры, играющие друг с другом; защиту решаем только
если/когда проект вырастет за пределы этого круга) — но закладываем модель прав на будущее:
**группы игроков** по аналогии с правами в Unix, что позволяет хранить права на объект в одном
`short` (компактно, не отдельная система per-объект).

Sandbox-режим редактирования включается **хостом**: хост в любой момент может переключить
в режим песочницы себя и опционально других игроков, и обратно. Отдельная защита от
злоупотреблений не нужна — хост это фактически админ сессии, игра не рассчитана на вечно
работающие сервера без присмотра, так что админ всегда на связи.

## Жизненный цикл сессии: GameManager / NetworkManager / GameState

Разделение ответственности (по мотивам UE GameMode/GameState, адаптировано под наш стек):

- **`GameManager`** — host-only "мозг матча", НЕ networked-объект сам по себе (как
  `AIController`/`AIDirector` — клиентская копия не нужна). Отвечает за жизненный цикл сессии:
  lobby → ready-up/countdown → игра идёт → (возможно) итоги. Решает, когда включены
  `AIController`/`AIDirector`, когда игрокам выдаётся боевой персонаж вместо призрака, разрешён
  ли переход в sandbox. Дирижирует `AIController`/`AIDirector`/`WorldManager`, не подменяет их.
- **`NetworkManager`** — только транспортные вещи: создание/поиск комнаты (Relay + Unity Lobby
  для списка комнат, см. раздел UI), обработка подключения/отключения клиентов, спавн
  `Player Prefab` (пара `PlayerState`+`PlayerController`) на каждое подключение — это то, что
  технически обязано происходить в момент коннекта в NGO. `GameState`/`AIController`/
  `AIDirector` — НЕ создаются на каждое подключение, это одиночные объекты на всю сессию,
  их создаёт `GameManager` один раз при старте хостинга.
- **`GameState`** — реплицируемая витрина того, что решает `GameManager`. `GameManager` пишет
  в `NetworkVariable`-ы `GameState` (фаза матча, время до конца countdown, число готовых
  игроков), клиенты читают `GameState` для UI. Плюс KV-контейнер для Lua-переменных уровня
  сессии (не привязанных к конкретному Entity).
- **`PlayerState`** — `NetworkObject`, по одному на подключившегося игрока.
- **`PlayerController`** — `NetworkObject`, но скрыт от всех клиентов кроме своего владельца
  через NGO network visibility (`NetworkShow`/`NetworkHide`) — у него в основном приватные для
  игрока данные (инпут, cooldown-ы абилок для UI и т.п.), другим игрокам это не нужно.

**Possession-паттерн (UE-стиль):** `PlayerController` не пересоздаётся при смене управляемого
персонажа — есть `Possess(Entity pawn)`/`Unpossess()`. Дефолтный призрак при входе → боевой
персонаж при старте матча → тот же (не новый) призрак при смерти, скрытый пока жив персонаж →
снова персонаж при респавне → золотой призрак при переключении в sandbox. `CameraOperator`
переключается вслед за текущей possessed Entity.

**Ready-up/countdown как скрипт:** MVP — жёстко закодированная логика в `GameManager`/
`GameState` (см. заметки на будущее ниже про более общий вариант со скриптуемыми UI-виджетами).

## Компоненты: Health, Spawner

- **`Health`** — текущее/максимальное здоровье + **типы урона** (по аналогии с Factorio —
  простая модель резистов/множителей по типу урона на цель), достаточно для разной прокачки
  игроков под разные типы врагов. Подсчёт фрагов/assist — не здесь (см. [[Разрешение
  одновременных killstreak/убийств]]), только сама механика здоровья/урона.
- **`Spawner`** — компонент на Entity: список тегов (чтобы `AIDirector` мог различать
  спавнеры), список юнитов/префабов на спавн, кулдаун между спавнами, флаг автоспавна, лимит
  спавнов (после которого автоспавн-флаг сбрасывается — точечное управление спавнером
  скриптом/`AIDirector`, например "заход игрока в зону → зональный скрипт командует спавнерам
  заспавнить 20 врагов с интервалом 2 секунды").

## Редактирование в sandbox не-хостом

Раз большинство компонентов не сетевые (`Health`, `Spawner`, `EntityVariables` и т.д.), а
`Entity` в общем случае не `NetworkObject` — правка конкретного компонента игроком, не
являющимся хостом, идёт тем же путём, что и любое другое игровое действие в этой архитектуре:
**клиент отправляет запрос на изменение хосту** ("установи такому-то `EntityId` такое-то
поле в такое-то значение"), **хост валидирует** (в перспективе — по модели прав/групп, см.
[[Права редактирования в sandbox]]) **и применяет**, затем рассылает применённое значение
наблюдателям через тот же кастомный event-канал. Ничего нового изобретать не пришлось —
редактирование в sandbox — просто ещё одна категория команд/событий в уже спроектированном
канале, наравне с атакой/гранатой/спавном.

## UI: структура экранов

- **`UIMainMenu`** — стартовый экран: "Создать" → `UICreateGame`, "Присоединиться" →
  `UIJoinGame`, "Настройки" → `UISettings`, "Выход".
- **`UICreateGame`** — создание комнаты (название, пароль, выбор мира).
- **`UIJoinGame`** — список доступных комнат + подключение по паролю. Список комнат — через
  **Unity Lobby** (сервис Unity под список комнат/join-by-code, естественно сочетается с
  Relay, который уже выбран для транспорта).
- **`UIInGameHUD`** — открывается после `UICreateGame`/`UIJoinGame`, пока идёт матч.
- **`UILoadingScreen`** — единственный UI-объект, активный сразу на старте (на стартовой
  сцене). Переключается видимостью (`SetActive(true|false)`) поверх остальных экранов на
  время загрузки мира между `UICreateGame`/`UIJoinGame` и `UIInGameHUD`.

## Синглплеер — не отдельный случай (подтверждено)

Раз архитектура host-authoritative, синглплеер — это просто сессия с одним игроком-хостом
без подключённых клиентов. Отдельного "оффлайн-режима" проектировать не нужно.

## Заметки на будущее (не решено, не блокирует текущую работу)

- **Прогрессивная активация областей мира.** Геймплей построен вокруг захвата новых областей
  карты — это естественный механизм ограничить активную симуляцию (незахваченные/нерелевантные
  области могут не думать/не реплицироваться), а не резать дальность симуляции искусственно.
  Стыкуется с решениями по производительности ИИ и по сетевой репликации. Прорабатывать при
  дизайне AIDirector (после исследования Rampant).
- **Мод-система шире, чем скрипты одного мира** (переиспользуемые контент-паки в духе
  Factorio-модов, а не только `Worlds/<name>/Scripts/`) — сознательно не планируется сейчас.
- **Дистрибуция через Steam** (и как следствие — Steam Workshop для передачи миров вместо
  своего bulk-transfer механизма) — сознательно не планируется сейчас, миры передаются
  вручную.
- **Скриптуемые UI-виджеты (Lua сам создаёт кнопки/таймеры на экране игрока).** Идея: скрипт
  на хосте мог бы сам инстанциировать виджет "готов" у всех игроков и тикать таймер обратного
  отсчёта, а не полагаться на жёсткую C#-логику `GameManager`/`GameState`. Архитектурно это
  укладывается в уже принятые паттерны (host-authoritative Lua, события в обе стороны через
  кастомный канал) и могло бы даже переиспользовать концепцию `Entity`/`EntityId` для самих
  UI-виджетов (виджет — тоже адресуемый по id объект с событиями, просто рендерится на Canvas,
  а не в 3D). Но это отдельная, довольно объёмная система (API спавна UI из Lua + маршрутизация
  кликов обратно в скрипт). **Рекомендация:** не строить её ради одного лишь ready-up —
  сначала сделать простой жёстко закодированный ready-up (см. [[Жизненный цикл сессии]]),
  чтобы быстрее дойти до играбельного прототипа, а "скриптуемые UI-виджеты" спроектировать
  позже как общую систему, если/когда понадобится не только для лобби.
